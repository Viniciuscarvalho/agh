# Analysis: Codex Sessions (~/.codex/sessions)

## Scope

- Source corpus: `~/.codex/sessions/2026/03/` (309 JSONL files, days 21–31, ~621 MB) and `~/.codex/sessions/2026/04/` (1,444 JSONL files, days 01–26, ~2.8 GB). Total available: 1,753 transcripts.
- Files read: ~35 sessions sampled across April with bias toward (a) the largest transcripts (>10 MB indicates extended autonomous runs), (b) sessions with the highest number of distinct user turns (top: 98, 71, 53, 49, 49, 46, 42, 41, 39 user messages — these are the genuine back-and-forth conversations), and (c) representative spot checks across late March 2026 to detect whether the patterns held before the AGH-only focus emerged. Late March was a different project (a Polymarket trading bot, `~/dev/projects/polybot`); from ~April 5 onward the sessions are dominated by AGH and the related Compozy/Looper repos.
- Aggregate Pedro-authored conversational text examined: ~7,350 distinct user-message lines (after stripping AGENTS.md preludes, environment context, network XML envelopes, and tmux-bridge envelopes).
- The transcripts are predominantly Brazilian Portuguese with English code/test names and skill invocations (Pedro routinely writes `$skill-name` references inline). Quotes below are reproduced verbatim — typos preserved.
- All session paths and IDs in this document are relative to `~/.codex/sessions/2026/`. File names follow `rollout-<UTC>-<UUIDv7>.jsonl`.

## Recurring Themes

1. **Skills are summoned by name on every non-trivial turn.** Pedro composes skills like a vocabulary — almost every implementation request opens or closes with `$systematic-debugging $no-workarounds`, often with `$golang-pro` or `$bubbletea` or `$refactoring-analysis` chained in. Examples:
   - `04/19/01-16-02`: "a responsividade da TUI está horrível, tudo lagado, nao respondendo em real time, o tempo da task nem sequer esta contando em segundos, streaming vindo errados sem fluidez, enfim bem ruim $systematic-debugging $no-workarounds $bubbletea ... faça um research para entender como integrar direito isso no $bubbletea ... $extreme-software-optimization"
   - `04/15/12-15-08`: "nós já mexemos 2x nisso e continuamos tendo que mexer, é mlehor fazer uma solução definitiva, por mais que seja demorada, do que ficar empurrando $no-workarounds $refactoring-analysis"
   - `04/18/19-27-52`: "use as skills golang, architectural analysis, no-workaround aqui tb"
     The skill chain is not decoration — Pedro expects the agent to actually load and follow them. When Codex misses a skill, Pedro re-issues the request with the skill prefixed.

2. **Subagents are exclusively for analysis, never for implementation.** This is enforced as a hard rule, made explicit on `04/24/11-27-50`:

   > `<critical>jamais use subagents para IMPLEMENTAR COISAS, sempre use eles só para analise</critical>`
   > Across many sessions the request shape is: "use multiplos subagents para explorar (apenas para explorer, modificacoes é sempre voce)" (`04/17/13-10-08`). When Pedro wants parallelism he wants it for _reading_ — reading the codebase, reading `.resources/<competitor>`, reading the conversation log saved in `~/.codex`, and producing analysis files (commonly under `.compozy/tasks/<name>/analysis/`).

3. **Cross-project research before architecture decisions.** Pedro routinely asks the agent to study three competitors at once (`.resources/claude-code`, `.resources/openclaw`, `.resources/hermes`, `.resources/openfang`, `.resources/multica`, `.resources/paperclip`, `.resources/goclaw`) before drafting any techspec.
   - `04/17/13-10-08`: "você precisa explorar o .resources/claude-code .resources/openclaw .resources/hermes .resources/openfang para entender como eles fazem para usar system prompts e execuções de agents customizados ... O que me parece é que, apesar de a gente ter um bom sistema determinístico aqui sendo feito, nós ainda estamos pecando bastante na parte de AI Harness"
   - `04/13/22-34-10`: "você vai chegar tnato os tópicos e os markdowns deles quando os .resources/\* que tem os repos atualizados com os arquivos reais"
   - `04/24/11-27-50`: "você tem que usar o .resources/hermes como referência para ver o que eles implementam no harness para suportar long running tasks com os agents"
     The pattern is: dispatch parallel subagents to write `analysis_<competitor>.md` files, then Pedro distills the findings, then `cy-create-techspec`.

4. **The PRD → TechSpec → Tasks pipeline is sacrosanct.** Almost every new feature run starts with `$cy-create-techspec` (sometimes preceded by `$brainstorming` or `$cy-create-prd`) and ends with `$cy-create-tasks`. The pattern Pedro enforces _every time_:
   - Before approving a techspec: run a peer review through Opus (`compozy exec --ide claude --model opus --reasoning-effort xhigh`) with sub-agents.
   - Append two final tasks (`$qa-report` + `$qa-execution`, sometimes with e2e) to the generated `_tasks.md`.
   - On `04/13/21-03-13`: "ok, mais além dessas 5 nós vamos fazer tipo o .compozy/tasks/harness e adicionar 2 novas tasks além que vão ser uma para o $qa-report e outra para o $qa-execution"
   - On `04/20/21-25-50`: "faça igual ao .compozy/tasks/agent-capabilities e adicione duas a mais no final para $qa-report e $qa-execution (com e2e cobrindo)"
     This shape is so consistent it could be a slash command.

5. **"Test coverage is fraco" is the canonical correction.** Pedro reads the generated `_tasks.md` and almost always pushes back on weak test plans:
   - `04/17/13-10-08`: "olhando as tasks geradas, me parece que o numero de tests unitarios e integration tb estão bem fracos, ou é impressão minha"
   - `04/17/18-02-53`: "o nivel de test unit e integration ficou fraco, melhore isso"
   - `04/18/23-48-40`: "aprovado, na hora de criar as tasks, você deve não ser leviano nos units e integration tests quando criar a lista deles"
   - `04/26/...`: "também é importante que cada task tenha uma boa cobertura de tests feitas corretamente, não ser leviano ou colocar apenas 1 ou 2 tests, mas fazer da forma correta $no-workarounds"

6. **"Zero mocks in final validation" — real software, real data.** When QA time comes, Pedro is explicit and harsh about avoiding fake fixtures:
   - `04/23/23-56-52`: "estamos prestes a fazer o release do nosso produto aqui ... a idéia é que você possa testar de fato o software, sem mock, sem stub, sem fake data, tudo da forma mais real possível e caso pegue issue, você vai fazer algo SIMILAR (não exatamente igual) ao $qa-report e $qa-execution, escrever issues e logo em seguida arrumar elas"
   - The same paragraph appears nearly verbatim in two other sessions (`04/22/...` and `04/24/...`) — it is a template.
   - The Codex base instructions Pedro accepts (visible in every `session_meta` record) mirror this: _"Mocks/stubs may exist in unit tests for I/O boundaries, but final validation must rely on real integration and end-to-end tests."_

7. **Memory ledgers and conversation continuity.** Pedro frequently asks Codex to re-read its own session log before continuing, citing `.codex/ledger/` files as the "compaction-safe" memory:
   - `04/17/13-10-08`: "use um subagent para conferir a conversa inteira nossa salva no estado em ~/.codex, pode ser com gpt 5.4 mini, mas ele vai verificar se não deixamos coisas para trás"
   - `04/18/23-48-40`: "antes use o gpt 5.4 mini high como subagent lendo o full log da conversa em ~/.codex para garantir que nada importante foi deixado pra trás e que nossa ledger está consistente com as decisoes"
   - `04/17/13-10-08` (after Codex forgot a decision): "c, só que já foi definido que não teríamos \_tasks e \_meta... viu, por isso falei pra você chegar o histórico da conversa para extrair as decisões, já está esquecendo coisas importantes"

8. **Web (`web/`) and docs (`packages/site`) are in scope of every backend change.** Pedro insists every backend feature checks for downstream UI/docs consequences:
   - `04/24/11-27-50`: "não é preciso mudar nada na UI do web/ por causa das mudanças aqui feita? e nem melhorar nada no packages/site na nossa documentação também?"
   - `04/26/...`: "em cada task tem que conter um subitem que é uma analise/implementação de mudanças necessárias decorrentes das alterações feitas na task nos packages web/ e packages/site... basicamente conferir se as mudanças feitas na tarefa implicam em mudanças na documentação e/ou no app web"
   - `04/25/22-42-35`: "eu acredito que nessa spec não está planejado quais partes do web/ e do packages/site (docs) vão precisar ser alteradas/melhoradas/adicionadas"

9. **Greenfield posture is repeatedly re-asserted.** When Codex slips into back-compat mode Pedro corrects immediately:
   - `04/15/12-15-08`: "esse projeto está em fase alpha de desenvolvimento, não tem ninguém usando, então não precisamos sacrificar arquitetura nele"
   - `04/17/13-10-08`: "esse é um projeto em desenvolvimento então podemos fazer coisas greenfield sem se preoocupar e não se preocupe com o tamanho do escopo disso"
   - `04/17/13-10-08`: "não ficou explicito nas tasks e na techspec que o modelo de desenvolvimento pode ser greenfield, que não precisamos adotar UM EXTREMO DE BACK COMPAT para tudo e gerar @deprecated stuff, ou ficou? isso não pode acabar gerando problemas se não formos mais explicitos sobre o nivel de greenfield vs back compat?"

10. **Single-source design references.** Whenever a UI task drifts, Pedro snaps it back to a specific image, brand kit, or competitor screenshot. Frequent triggers: "está totalmente errado, veja a imagem de referencia" (`04/24/17-18-43`), "está MUITO ruim em termos de pitch de vendas" (`04/15/...`), "muito complexo, deixe ele mais simples, crie um componente para cada card, sem abstrações desnecessárias" (`04/24/17-18-43`).

## Anti-Patterns Pedro Pushed Back On

- **Subagents writing code.** "jamais use subagents para IMPLEMENTAR COISAS, sempre use eles só para analise" (`04/24/11-27-50`). Reinforced after Codex tried to delegate edits in earlier turns.
- **Headless `compozy exec` opening a TUI.** `04/...`: "esse comando deveria ser mais headless, o --format text|json deveria falar sobre como o output é feito, quando json todas as resposta deveriam ser JSON normais, não só um ultimo update em json, e quando text, deveria ser uma única resposta em string no final, não abrir a TUI". Pedro rejected the auto-creation of `.compozy/runs` artifacts when `exec` is invoked: "também não deveríamos ter que sempre persistir em .compozy/runs para ele ter que rodar".
- **Solutions that reuse the wrong abstraction twice.** `04/15/12-15-08`: "nós já mexemos 2x nisso e continuamos tendo que mexer, é mlehor fazer uma solução definitiva, por mais que seja demorada, do que ficar empurrando". When something has been patched twice already, Pedro forbids further patches and demands the structural fix.
- **Generic AI design ("AI slop" cards / icons / dark-mode default).** `04/24/14-20-02`: "essa imagem não vende nada, é extremamente complexa e com muita informação". `04/...`: "crie uma imagem usando $imagegen para redesenhar essa parte da nossa landing page para algo mais atrativo usando ilustrações nos cards para sairmos dessa padrão já batido de cards com icones e título". Pedro is explicitly resistant to default Tailwind/shadcn looks and "boring cards-with-icons" patterns.
- **`@deprecated` stubs and migration shims.** Per the greenfield theme — "sem deprecated e tudo rodando como deveria bem arquiteturado" (`04/15/12-15-08`).
- **Faux-human checkpoint loops.** When Codex paused waiting for confirmation, Pedro corrected in the orchestrator skills: "Do not stop waiting for a human approval loop. Assume the orchestrator's reply is authoritative for PRD, TechSpec, and task-breakdown decisions" (`04/18/19-53-56`, repeated across smux pairing prompts). Pedro engineered around the confirmation-pause behavior by writing the orchestrator role to act as the decider.
- **Leaking implementation choices into PRDs.** Across the `todo-api` smux pairing run, the Claude pane is instructed to reject any PRD that surfaces frameworks/storage/error codes: "PRD naming frameworks/storage engines/file formats — strip, push to TechSpec." Pedro accepted this directly and adopted it as the rule.
- **Fixture-substring matching in E2E tests.** Pedro rejected fragile substring-based assertions in E2E suites and demanded structural matchers (cross-referenced in plans like `2026-04-17-e2e-confidence-hardening.md`).
- **Hardcoded ports / hardcoded paths.** `.compozy/tasks` was hardcoded across many flows; Pedro insisted on a `config.toml` plus discovery: "para evitar workaround e não nascer com escopo ruim, preciso travar uma decisão de produto primeiro" (`04/...` polybot config session, with the same phrasing later in AGH config decisions).
- **Behavior changes that silently break interactive UX.** `04/20/10-04-04`: a CodeRabbit review flagged that flag-marking turned cleared optional fields into explicit overrides; Pedro told Codex "fix this one $no-workarounds". The pattern: even small behavior regressions in interactive paths get the no-workarounds skill explicitly attached.
- **TUI quitting auto-stops the session.** `04/...`: "nós introduzimos recentemente um behavior que se o usuário está numa run e da quit dessa run, automaticamente a run da stop, mas isso foi uma decisão errada". The original implementation was reverted because the UX assumption was wrong.
- **Boolean-prop proliferation in UI components.** `04/24/17-18-43`: "o código do bento-section.tsx está muito complexo, deixe ele mais simples, crie um componente para cada card, sem abstrações desnecessárias". Pedro consistently prefers many small components over generic composable monsters.
- **Over-thinking the data model.** `04/19/...`: "isso é over thinking, o agente pode colocar o que ele quiser" (about whether to require structured fields in the agent capabilities format).
- **Generating new conventions when repo conventions exist.** `04/...`: "vamos seguir o mesmo padrão de endpoints do AGH, rest endpoints, nada de inventar, e vamos usar o mesmo framework web do AGH" — when daemonifying Compozy, Pedro mandated copying AGH's REST conventions verbatim rather than redesigning.

## Engineering Principles Pedro Enforces

1. **Definitive solutions over incremental patches once a code area is touched twice.** Evidence: "nós já mexemos 2x nisso e continuamos tendo que mexer, é mlehor fazer uma solução definitiva". After two patches, the rule is structural redesign.
2. **Architecture is decided up front; implementation flexibility comes later.** Evidence: "nada de fazer algo meia boca em v1, vamos ja pensasr no modelo definitio" (`04/18/23-48-40`). Pedro will hold up a techspec until the v1 design is the definitive one.
3. **Spec quality is measured by absence of open questions, not length.** Evidence: "temos que decidir, uma spec ja tem que ir com a versao final decidiade para depois ser só implementada, nao deixar aberto para duvidas" (`04/17/18-02-53`).
4. **Cross-reference competitors before designing — and link the references in the task files.** Evidence: "também é importante que cada task tenha referências dos projetos do .resources/\* como temos na techspec, é bom colocar, pois isso faz olharmos antes esses arquivos para pegar referencias e inspiracao". Pedro wants paths in the task body so the implementation agent reads them too.
5. **Explore before designing, but the final author is always Codex (the human-paired agent), not a subagent.** Evidence: "use multiplos subagents aqui para explorar (apenas para explorer, modificacoes é sempre voce)".
6. **The right tool for the right format: A/B/C menus for design choices, single-option confirmations for already-decided constraints.** Evidence — from his orchestrator prompt template: "If a decision is already constrained by approved PRD answers, accepted ADRs, or explicit non-goals, carry it forward directly or ask for a single-option confirmation. Do not reopen it as a fresh A/B/C/D menu." Pedro pushes back when Codex re-opens settled questions.
7. **Branding consistency across surfaces.** Evidence: when generating illustrations, "siga o estilo das imagens em /Users/pedronauck/Dev/compozy/agh/packages/site/public/images/bento-illustrations" — even subtle palette mismatches get rejected.
8. **No daemon should require a foreground terminal.** Evidence: "ao executar 'compozy daemon start', assim como no ~/dev/compozy/agh, o daemon deveria rodar em background, não um processo que tem que ficar com a aba do terminal aberta" (`04/18/23-46-05`). This is repeated for every CLI surface — Pedro treats backgrounded daemons as a baseline requirement.
9. **Default UX must be idiomatic for the surface.** "os comandos que tinham TUI neles ... deveria por padrão quando não fornecido argumentos, sempre abrir a TUI, para manter o padrão de DX". A bare command opens the TUI; flags are needed to suppress it. Default behaviors must match user habits.
10. **Hot-reload affordance for development is a first-class concern, but kept simple.** From `04/21/12-03-34`: after a back-and-forth where Codex proposed orchestrating Vite + a Go-side Air rebuild, Pedro stops it: "isso ainda ta muito complexo, vamos fazer assim... deixa no make file só o ./bin/compozy com uma flag, e eu rodo em um outro terminal o vite dev dai". Local DX wins over orchestration cleverness.
11. **Sandbox isolation when multiple agents run in parallel worktrees.** `04/...` (multiple sessions): "$qa-execution .compozy/tasks/core-tasks (você deve isolar o AGH_HOME aqui e a porta, pois temos outro agent testando outra worktree e não pode haver conflitos entre voces dois)". Pedro routinely runs concurrent Codex sessions and demands explicit isolation.
12. **Resource cost is a hard limit, not an aspiration.** `04/18/11-54-10`: after a runaway memory event, "eu tive que reiniciar o computador porque estourou totalmente CPU/MEMORIA e travou tudo aqui ... se for rodar, tente rodar em algum docker ou algum ambiente que você possa controlar o uso para não acabar com tudo aqui". Resource-hungry test loops or compaction passes are a violation.

## Skill Candidates (new skills worth creating)

1. **`research-competitors`** (or extend `cy-create-techspec`).
   - Trigger: any techspec or large refactor task whose `_idea.md` references `.resources/<repo>/` or names a competitor (Hermes, OpenClaw, Multica, Paperclip, Goclaw, Claude Code, Codex CLI).
   - Behavior: dispatch parallel read-only subagents (gpt-5.4-mini-high is Pedro's preference for breadth; gpt-5.4-xhigh for depth — explicitly asked for in `04/19/23-24-15` and `04/13/22-34-10`), each writing `.compozy/tasks/<slug>/analysis/analysis_<competitor>.md` with sections: Overview, Mechanisms/Patterns, Relevant Code Paths, Transferable Patterns, Risks/Mismatches, Open Questions, Evidence (file paths).
   - Evidence: this exact shape was hand-orchestrated in at least 6 sessions (`04/13/22-34-10`, `04/17/13-10-08`, `04/17/18-02-53`, `04/18/23-48-40`, `04/24/11-27-50`, `04/25/22-42-35`). The competitor-comparison phase is always the same — turn it into a skill.

2. **`spec-final-review`** (peer-LLM cross-check).
   - Trigger: a techspec has been written and is about to be approved.
   - Behavior: invoke `compozy exec --ide claude --model opus --reasoning-effort xhigh --format json` with a curated prompt that asks Opus to identify "blocking issues, non-blocking nits, and readiness". The prompt template is already written by Pedro in many sessions.
   - Evidence: "antes de aprovar, você vai usar o $compozy exec --ide claude --model opus --reasoning-effort xhigh --format json para revisar sua spec final" (`04/20/21-25-50`); "agora você vai usar o $compozy para pedir ao exec --ide claude <prompt> ... e fazer a revisão final dele encima da sua spec" (`04/17/13-10-08`); "ok, rode mais um review do opus agora, principalmente para checar todos os docs e partes para achar gaps e inconsistencias que possam prejudicar o trabalho final" (`04/25/22-42-35`). Pedro does this cross-LLM review every single time.

3. **`task-tail-qa-pair`** (auto-append `qa-report` + `qa-execution` tasks).
   - Trigger: after `cy-create-tasks` finishes generating `_tasks.md`.
   - Behavior: append two final tasks mirroring the structure used in `.compozy/tasks/agent-capabilities`, `.compozy/tasks/daemon`, `.compozy/tasks/hermes`, `.compozy/tasks/unified-capabilities` (these are the named templates Pedro cites). The tasks must include e2e where the feature has user-facing surface.
   - Evidence: this is the single most repeated request — appears in nearly every techspec session reviewed. "faça igual ao .compozy/tasks/agent-capabilities e adicione duas a mais no final para $qa-report e $qa-execution (com e2e cobrindo)" (`04/20/21-25-50`); "ok, agora vamos $cy-create-tasks ... ok, faça essa e mais 2 como no .compozy/tasks/hermes que serão usadas para $qa-report $qa-execution" (`04/26/...`).

4. **`web-docs-impact`** (UI/docs ripple-check on every backend change).
   - Trigger: a backend task is being implemented or a techspec is being drafted.
   - Behavior: after the backend plan is settled, run a pass over `web/` and `packages/site/` to identify which routes, components, hooks, or doc pages are affected, and either generate an additional task or annotate the parent task with a "web/docs impact" subitem.
   - Evidence: "em cada task tem que conter um subitem que é uma analise/implementação de mudanças necessárias decorrentes das alterações feitas na task nos packages web/ e packages/site"; "não é preciso mudar nada na UI do web/ por causa das mudanças aqui feita? e nem melhorar nada no packages/site na nossa documentação também?"; "eu acredito que nessa spec não está planejado quais partes do web/ e do packages/site (docs) vão precisar ser alteradas/melhoradas/adicionadas". This is asked literally every time, suggesting the agent should not need to be reminded.

5. **`worktree-isolation`** (parallel-agent safety).
   - Trigger: a QA execution or test run is being prepared while the user has signaled (via env or task argument) that other agents are also running.
   - Behavior: enforce per-worktree `AGH_HOME`, unique daemon ports, and unique `tmux-bridge` socket paths. Block any operation that would write to `~/.agh/` or default ports.
   - Evidence: "$qa-execution .compozy/tasks/core-tasks (você deve isolar o AGH_HOME aqui e a porta, pois temos outro agent testando outra worktree e não pode haver conflitos entre voces dois)" — repeated across sessions in similar wording.

6. **`tasks-test-pressure`** (test-coverage stress-tester).
   - Trigger: after `cy-create-tasks` produces `_tasks.md` or a single `task_NN.md`.
   - Behavior: read the generated test plan, count assertions vs. behaviors documented in the techspec, and emit a critique: "you have N tests for M behaviors; M-N behaviors lack coverage; reject as `fraco`". This is exactly Pedro's manual review pass.
   - Evidence: "olhando as tasks geradas, me parece que o numero de tests unitarios e integration tb estão bem fracos, ou é impressão minha"; "o nivel de test unit e integration ficou fraco, melhore isso"; "aprovado, na hora de criar as tasks, você deve não ser leviano nos units e integration tests".

7. **`real-scenario-qa`** (already exists in this repo as `.agents/skills/real-scenario-qa/`, untracked at the time of this analysis — confirms the candidate above).
   - The existing `real-scenario-qa/` skill folder was added at the same time as the autonomous QA work. The session-level evidence suggests it was distilled from the recurring pattern: "testar de fato o software, sem mock, sem stub, sem fake data". It already covers what would otherwise be a separate skill candidate.

## Lesson-Learned Candidates (one-off insights worth capturing)

- **The "mexemos 2x" rule.** When the same code area is touched twice in the same workstream, the third change must be a structural redesign, not a third patch. The cost of a redesign is paid once; the cost of a third patch compounds. Source: `04/15/12-15-08`, where this principle was applied to the `internal/extension` extensibility refactor.
- **Backgrounded daemons are a hard requirement, not a feature.** When daemonifying Compozy, Pedro listed five concrete pains caused by execution-mode-only operation: "failover mais dificil de ser gerenciado / dificil de colocarmos uma segunda interface, por exemplo, web para isso / dificil de através de extensões..." — these are the symptoms a techspec should call out when justifying daemonization (`04/17/13-10-08`).
- **`config.toml` over flag-everything.** Multiple sessions converged on the same conclusion: persisting workspace defaults in `.compozy/config.toml` (or `~/.compozy/config.toml`) prevents the CLI flag-spread that was happening across `start`, `fix-reviews`, and `exec`. The decision tree: defaults → workspace conventions → per-flag-scope-override. Source: `2026-03-31` ledger snapshot embedded in an early session.
- **Boundary surfaces collide; spec authors must reconcile them.** When two specs are drafted in parallel (`extensibility-parity` and `sandbox` on `04/15`), the later one almost certainly has interference points with the earlier one in shared packages — `internal/extension`, `internal/session`, `internal/api/*`, `internal/config`, `internal/workspace`, hooks, host API, daemon composition. Pedro caught this by asking for an explicit interference audit between specs. Lesson: when two techspecs are drafted within a 2-week window, an "interference audit" task should be inserted between them.
- **TUI quit ≠ run stop.** Auto-stopping a run when the user exits the TUI was implemented and immediately reverted. The right shape is a confirmation modal: "ver com o $bubbletea quando ele der quit como mostrar algo como um modal perguntando se ele quer stopar a run ou não" (`04/...`). Lesson: dialog-tier UX decisions ("close == cancel?") are bug-likely if implemented as auto-behaviors.
- **`compozy archive` failed silently because of a "sync state" check.** `04/21/13-14-57`: archiving was blocked by `globaldb: workflow "agent-capabilities" is not archivable: workflow state not synced (request_id=req-...)`. The bug was that the daemon required the workflow's SQLite-projected state to be synced before allowing archive — but workflows that finished cleanly never re-synced. Lesson: archive-on-completion paths must not depend on a separately-synced projection.
- **TUI lag is almost never a TUI bug.** `04/19/01-16-02`: "a responsividade da TUI está horrível, tudo lagado, nao respondendo em real time, o tempo da task nem sequer esta contando em segundos". The fix lived in the streaming/event-fan-out path, not in `bubbletea` rendering. Lesson: when TUI feels slow, suspect the producer side first.
- **Subprocess shutdown races between health monitor and stop path.** `04/15/19-51-54` (subprocess subagent) found a race where `stopHealthMonitor()` cleared a channel field while the monitor goroutine was reading it. Fix: per-run immutable struct (`healthMonitorRun`) so the goroutine owns its lifecycle. Lesson: never put goroutine-owned channels in a struct field that another goroutine mutates — use a per-run handle.
- **`DecodeSSE` shared decoder swallows `ErrStop`.** `04/15/19-51-54` (CLI subagent): the shared SSE decoder treats `ErrStop` as success and keeps consuming the stream, but the CLI uses `errStopSSE` as a "stop now" signal. Fix: do not share a decoder between transports with different stop semantics. Lesson: SSE/streaming decoders should be parametric on stop-signal handling, not assume one.
- **AtomicWriteFile assumes parent dirs exist.** `04/15/19-51-54` (telegram-reference subagent) found markers silently failing because their parent directories didn't exist. Lesson: any "atomic write" helper must explicitly accept a `mkdirIfMissing` policy.
- **Sub-slide ordering.** A non-AGH session (slide deck export, `04/...`) hit the "data-appear ordering is derived from DOM order, not file order" bug. The fix: explicit subslide numbering `10/1, 10/2`. Lesson generalizes: when you derive an order from a presentation-time state, you lose authorial intent — preserve the source order separately.

## Surprising / Non-Obvious Findings

- **Pedro is running a multi-LLM development pipeline as a routine.** Almost every session involves Codex (gpt-5.4-xhigh) writing the spec, Claude Opus reviewing it, sometimes a third LLM (gpt-5.4-mini-high) doing parallel exploration. This is not a one-off — it is the default workflow. The repository's `.compozy/` artifacts are designed around this multi-LLM dance. CLAUDE.md doesn't currently capture that this is the _expected_ dev model.
- **The `tmux-bridge` + `smux-pair` workflow exists as a hand-orchestrated multi-agent pairing.** On `04/18/19-27-52`, Pedro hand-built a skill so two agents (`todo-api-claude` and `todo-api-codex`) could pair-program through tmux panes — with a third orchestrator agent reading both panes and resolving checkpoints. This is the seed of what would later become the autonomous orchestration features. The skill's design constraint is precisely "no human in the loop" — Pedro built tooling to remove himself from approval cycles.
- **The autonomous review batch (`reviews-001/`) is a workflow product.** The session at `04/26/00-50-43` is a `cy-fix-reviews` invocation against a CodeRabbit-generated batch of 25 issues. The dispatch input is structured XML with the entire batch scope. This means there's a programmatic pipeline producing these — not Pedro typing each issue.
- **Pedro's pushback language has consistent phonetic markers.** "fraco" (weak), "leviano" (lazy/lightweight), "ruim" (bad), "está totalmente errado" (totally wrong), "meia boca" (half-assed), "esquecendo coisas" (forgetting things). When these words appear, the agent has typically just produced something Pedro considers under-engineered. They function as escalation signals — the appearance of `meia boca` means "redo with no compromise".
- **`<turn_aborted>` is a frequent companion to corrections.** Pedro often interrupts mid-execution and immediately re-issues the request with more skills attached. This implies he reads the agent's work in real-time and aborts as soon as the trajectory looks wrong. The pattern "<turn_aborted>...$systematic-debugging $no-workarounds" appears dozens of times. The agent should treat repeated aborts in the same session as a strong signal to slow down and re-clarify before acting.
- **The polybot sessions in late March are about a different project but the engineering-discipline language is identical.** Same skills (`$systematic-debugging $no-workarounds`), same insistence on "100% operacional e autonomo, sem mock, sem stub", same intolerance for partial fixes. This suggests these patterns are Pedro's general engineering posture, not AGH-specific. Anything in CLAUDE.md framed as "AGH-specific" that actually generalizes (e.g., the no-mock-in-final-validation rule) could be elevated.
- **Parallel agent runs are normal in worktrees.** Multiple sessions explicitly mention "tem outro agent trabalhando em arquivos em tempo real aqui, foque nas duas specs, deixa esses errors pra ele" (`04/13/22-34-10`) and "tem outro agent testando outra worktree". Pedro routinely runs 3+ Codex sessions in parallel against different `_worktrees/`. The system prompt should anticipate this.
- **Many sessions are short (1–3 user turns) because Pedro pre-loaded everything in the first message.** The short-session pattern reveals Pedro often constructs huge structured prompts that include the techspec reference, the issue list, the skills to load, and the output contract — then expects a single complete-end-to-end response. The agent's challenge is that these prompts often span 15-20 KB and look like XML. The "<arguments>...<critical>...<execution_contract>" pattern is the signal that this is autonomous-mode dispatch, not interactive.
- **CodeRabbit P2-style issues get the same treatment as P0s.** On `04/20/10-04-04`, a P2 issue about flag marking on cleared form fields was attacked with `$no-workarounds` — same skill stack as critical bugs. Pedro doesn't tier-rank his fix discipline by issue severity.
- **The brand book and design discipline are first-class concerns.** Sessions on `04/22/13-13-04` and `04/24/14-20-02` show Pedro generating brand-kit images, logo variations, illustrations — all explicitly tied to `DESIGN.md` constraints (no shadows, warm-dark, the orange accent, etc.). UI/marketing work is not an afterthought; it gets the same skill rigor as backend work, with `$design-taste-frontend $minimalist-ui $no-workarounds` as the standard chain.
- **Codex's `apply_patch` requirement bites occasionally.** Several sessions have the exact warning "Warning: apply_patch was requested via exec_command. Use the apply_patch tool instead of exec_command." appearing as a mid-turn message. This is Codex's tool harness correcting itself, not a user message. It indicates Codex sometimes tries to write files via shell instead of the patch tool.
