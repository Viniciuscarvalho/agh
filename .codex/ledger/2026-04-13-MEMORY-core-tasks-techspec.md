Initial clarification decisions already accepted: `Task`/`TaskRun` split; `global + workspace` scope; explicit `parent_task_id` hierarchy plus dependency edges; wide writer set (`task`, `automation`, `extensions`, `network peers`); canonical `Task.status` owned only by `TaskManager`; queue-first `TaskRun`; optional channel binding captured in ADR-004. Post-review redesign decision: tasks are an optional explicit resource, not the universal representation of every agent action. Task creation may come from explicit user intent, explicit system configuration, or explicit agent intent, but not from silent heuristic inference. Authorization model direction from user: `created_by` must remain fixed and record whether creation came from a human or an agent; `owner` is optional at creation time, may be auto-assigned by the system, may remain unowned/pool-backed, and may later change; `origin` should exist as separate immutable technical ingress metadata. Initial authorization policy direction: in v1 any authenticated/authorized writer surface may create both `global` and `workspace` tasks, but actor identity must still be derived server-side rather than trusted from payload. Mutation policy direction from user acceptance: `created_by` and `origin` are immutable; `owner` may be reassigned by any authorized writer; structural fields such as `scope`, `workspace_id`, and `parent_task_id` are immutable after creation; editable fields include `title`, `description`, `metadata`, and channel binding; cancellation is allowed to any authorized writer; canonical completion remains under `TaskManager` control rather than arbitrary patching. Execution model direction from user: tasks themselves do not create sessions; executable subtasks are the default execution unit, and when a subtask starts it should, by default, receive a dedicated new session. Attaching an existing session is reserved for explicit continuation, resume, or handoff cases. Cancellation model direction from user: parent-task cancellation should propagate through the task tree, queued work should cancel immediately, active subtask sessions should receive cooperative stop first, and forced stop should be used only after a grace window configured globally at the daemon level. Automation/task boundary direction from user: automations should support both paths for task creation in v1: direct task-service writes by automation itself and prompt-driven agent sessions that explicitly call `task.create`. Prompt text alone must still not cause implicit runtime task creation. Payload hardening direction: cap `metadata_json` at 16 KB and cap both `payload_json` and `result_json` at 64 KB; larger data should live in separate artifacts referenced by task/run records. Dependency graph hardening direction: detect cycles on edge creation, cap tree depth at 8, cap dependency edges per task at 32, and cap direct children per task at 64. Package-boundary direction from user: `internal/task` should not import `internal/session` directly; session creation/attach/stop behavior should come through a small injected interface implemented at daemon wiring time.
