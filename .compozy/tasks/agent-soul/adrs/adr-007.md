# ADR-007: Treat HEARTBEAT.md as Advisory Wake Policy

## Status

Accepted

## Date

2026-05-01

## Context

`HEARTBEAT.md` has overloaded meanings in reference systems. OpenClaw uses it to shape periodic agent turns, Paperclip uses it as adapter-specific loop instructions, and Hermes uses heartbeat mainly for runtime liveness and durable run ownership.

AGH already has distinct heartbeat authorities: task-run lease heartbeat, session activity supervision, scheduler wake/sweep, synthetic prompt reentry, and AGH Network greet presence. Introducing `HEARTBEAT.md` as another heartbeat authority would confuse ownership, liveness, and scheduling.

## Decision

`HEARTBEAT.md` v1 is an optional authored wake-policy artifact. It may influence synthetic wake/reentry guidance, prompt summary, and policy evaluation for already eligible sessions. It must not create work, create sessions, auto-spawn agents, run an independent loop, renew task-run leases, update session activity, or alter network peer presence.

Executable work remains represented by `task_runs` and acquired only through `ClaimNextRun`.

## Alternatives Considered

### Alternative 1: Auto-Spawn Heartbeat Sessions

- **Description**: Let `HEARTBEAT.md` trigger daemon-created heartbeat sessions when no session is active.
- **Pros**: Closer to Paperclip/OpenClaw behavior and higher autonomy.
- **Cons**: Expands lifecycle, cost, loop control, and ownership risk.
- **Why rejected**: V1 should not create a new execution authority.

### Alternative 2: Materialize Heartbeat as Task Runs

- **Description**: Convert due heartbeats into `task_runs`.
- **Pros**: Strong durability and claim semantics.
- **Cons**: Pollutes the durable work queue with self-checks and turns Markdown into a scheduler.
- **Why rejected**: The approved MVP is advisory wake policy, not recurring work execution.

## Consequences

### Positive

- Preserves `task_runs` and `ClaimNextRun` as the only executable work path.
- Keeps liveness and authored wake policy separate.
- Allows a useful `HEARTBEAT.md` artifact without hidden workflow semantics.

### Negative

- V1 does not autonomously create new sessions when no eligible session exists.
- Users expecting OpenClaw-style periodic model turns may need a later feature.

### Risks

- Operators may still misread "heartbeat" as liveness; API names and docs must consistently use wake-policy language.

## Implementation Notes

- Use implementation names like `AgentHeartbeatPolicy`, `ResolvedHeartbeatPolicy`, and `WakePolicy` where clarity matters.
- Manual wake returns typed diagnostics when no healthy idle session is eligible.
- The scheduler may consult the policy only after normal eligibility checks.
