# ADR-005: Defer HEARTBEAT.md to a Separate Runtime Policy Spec

## Status

Accepted

## Date

2026-05-01

## Context

`SOUL.md` describes who an agent is and how it behaves. `HEARTBEAT.md` describes what an agent should do when it wakes, how it handles recurring checks, and how heartbeat-driven work should run. Competitor systems show `HEARTBEAT.md` quickly crosses into scheduling, wake coalescing, liveness classification, task ownership, API calls, and model-driven background turns.

AGH already has separate authorities for task-run lease heartbeat, session activity heartbeat, scheduler sweep/notify/recovery, and AGH Network greet presence. Mixing `HEARTBEAT.md` into the Agent Soul feature risks creating a hidden workflow engine in Markdown or bypassing `ClaimNextRun`.

## Decision

`HEARTBEAT.md` is explicitly out of scope for the Agent Soul TechSpec. The Agent Soul feature will not implement recurring model-driven heartbeat turns, heartbeat task parsing, wake scheduling, task queue behavior, lease heartbeat changes, scheduler claim behavior, or AGH Network presence changes.

Any future `HEARTBEAT.md` or wake policy work must be specified separately and must preserve `task_runs` and `ClaimNextRun` as the authoritative executable work and claim path.

The future work anchor is a TechSpec slug to be created after Agent Soul ships: `agent-heartbeat`. It must not start until a wake-policy PRD exists.

## Alternatives Considered

### Alternative 1: Implement HEARTBEAT.md Together with SOUL.md

- **Description**: Add both persona and operational heartbeat files in one feature.
- **Pros**: Matches some competitor file sets.
- **Cons**: Mixes identity authoring with runtime scheduling and liveness.
- **Why rejected**: It expands scope into orchestration and risks a second task system.

### Alternative 2: Add HEARTBEAT.md as Prompt-Only Guidance

- **Description**: Let `HEARTBEAT.md` be a non-authoritative checklist in prompt context.
- **Pros**: Lower implementation cost.
- **Cons**: Ambiguous authority; agents may treat it as operational truth without runtime support.
- **Why rejected**: Operational wake behavior needs typed AGH primitives.

## Consequences

### Positive

- Keeps Agent Soul focused on persona and identity.
- Protects existing heartbeat, scheduler, task, and network authorities.
- Leaves room for a stronger future wake policy design.

### Negative

- Users do not get Paperclip/OpenClaw-style heartbeat checklist files in this feature.

### Risks

- The future work may be expected soon because competitor systems expose `HEARTBEAT.md`; the TechSpec must name it explicitly as deferred, not forgotten.

## Implementation Notes

- Mention `HEARTBEAT.md` only in Out of Scope and Future Work sections.
- Future work should model operational wake policy through existing task and scheduler primitives.
- Use `agent-heartbeat` as the working slug when that separate PRD/TechSpec is opened.
- 2026-05-01 update: the separate Heartbeat TechSpec is now being drafted from direct user-provided context under `.compozy/tasks/agent-soul/_techspec_heartbeat.md`; this ADR remains the boundary that kept Heartbeat out of the Soul TechSpec.
