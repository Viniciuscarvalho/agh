# ADR-010: Provide Managed Heartbeat and Session Health Surfaces in MVP

## Status

Accepted

## Date

2026-05-01

## Context

AGH features are incomplete if they are only internal Go calls or web UI controls. Agent Soul already chose managed authoring with CLI, HTTP, UDS, and Host API parity. `HEARTBEAT.md` depends on session health to be useful, so both the artifact and runtime health state need agent-operable surfaces.

## Decision

MVP manageability covers both `HEARTBEAT.md` policy and session health:

- `agent heartbeat` surfaces manage the authored artifact and resolved policy.
- `session status`, `session health`, and `session inspect` surfaces expose runtime session health and wake eligibility.
- CLI, HTTP, UDS, OpenAPI, generated TypeScript, and Host API actions co-ship where applicable.

The API must not expose raw claim tokens or sensitive prompt/provider data.

There is no explicit heartbeat refresh surface in MVP. Policy writes create new snapshots, and wake decisions evaluate the latest valid snapshot at wake-decision time. The snapshot id and digest used for a wake are recorded on the wake event and synthetic prompt metadata.

Managed authoring CAS uses the request body field `expected_digest` across HTTP, UDS, CLI, and Host API. CLI `--if-match` maps to that field. HTTP `If-Match` headers are rejected with `heartbeat_if_match_header_unsupported`.

## Alternatives Considered

### Alternative 1: Manage HEARTBEAT.md Only

- **Description**: Add policy authoring surfaces but keep session health internal.
- **Pros**: Smaller public API.
- **Cons**: Agents cannot diagnose wake eligibility.
- **Why rejected**: Wake policy depends on health; hiding health weakens manageability.

### Alternative 2: Internal Health Plus Read-Only HEARTBEAT.md

- **Description**: Add health primitive, but keep authoring manual.
- **Pros**: Conservative runtime rollout.
- **Cons**: Inconsistent with Agent Soul's managed authoring direction.
- **Why rejected**: The user approved managed authoring parity.

## Consequences

### Positive

- Agents can inspect, configure, operate, and debug the feature without web UI access.
- Operators can distinguish invalid policy, no eligible session, rate limiting, and runtime health failure.

### Negative

- Larger contract/codegen/docs surface in v1.

### Risks

- Surface drift across CLI/HTTP/UDS would undermine trust; parity tests are mandatory.

## Implementation Notes

- `agh agent heartbeat inspect|validate|write|delete|history|rollback|status|wake`.
- `agh session status|health|inspect`.
- Deterministic errors include `heartbeat_disabled`, `heartbeat_invalid`, `heartbeat_conflict`, `heartbeat_if_match_header_unsupported`, `heartbeat_rate_limited`, `heartbeat_no_eligible_session`, `heartbeat_no_policy`, `session_unhealthy`, `session_not_attachable`, `session_prompt_active`, `session_prompt_active_race`, and `session_not_found`.
- Parity tests must cover HTTP, UDS, CLI, and Host API DTO shape for `expected_digest`, wake result, status, history, rollback, and deterministic errors.
