# ADR-003: Snapshot Resolved Soul at Session Start and Task Claim

## Status

Accepted

## Date

2026-05-01

## Context

Long-running sessions need deterministic persona behavior. Hermes stores a resolved system prompt per session, avoiding surprise mid-session changes. Paperclip's sibling-file approach can leave persona changes ambiguous across resumed sessions. AGH also needs task-run auditability: a run should record which resolved persona digest was active when the agent claimed work.

## Decision

AGH will resolve and validate `SOUL.md` during agent profile loading. It will snapshot the resolved soul digest/profile at session start and snapshot the active digest/profile again at task-run claim time. AGH will not silently reload `SOUL.md` during active sessions or active task runs.

Changes to `SOUL.md` take effect for new sessions and new task-run claims. Existing sessions or runs can refresh only through an explicit reload/refresh operation.

Task-run claim provenance is copied from the claiming session's already-persisted `sessions.soul_snapshot_id` and `sessions.soul_digest`. `ClaimNextRun` performs no `SOUL.md` resolution, parsing, validation, digesting, path traversal, or filesystem I/O. Claims without a session snapshot omit the persisted `soul` metadata block and use a typed internal diagnostic sentinel instead of resolving from disk.

Explicit refresh validates/parses under the request context, then performs durable snapshot upsert, session row update, and audit event emission under a detached bounded context. Refresh and session task claim share a session-scoped lock so refresh returns `409` instead of racing an active or starting claim.

## Alternatives Considered

### Alternative 1: Reload on Every Prompt

- **Description**: Read and apply `SOUL.md` before every model call.
- **Pros**: Edits take effect immediately.
- **Cons**: Causes invisible behavior drift and weakens auditability.
- **Why rejected**: Long-running AGH sessions require reproducible context.

### Alternative 2: Snapshot Only at Session Start

- **Description**: Record the soul digest only when a session starts.
- **Pros**: Simple and stable.
- **Cons**: Task runs need their own audit trail, especially when claimed by resumed sessions.
- **Why rejected**: Run-level provenance matters for task execution and reviews.

## Consequences

### Positive

- Prevents silent mid-run persona changes.
- Gives task-run audit trails a stable persona digest.
- Allows explicit refresh when an operator or agent wants new persona content.

### Negative

- Users may need to restart or refresh sessions to apply changes.
- Requires storing digest/profile references in session and task-run metadata.

### Risks

- Storing the full body on every run can bloat storage; store digest plus bounded snapshot/provenance and keep full resolved read model available separately.

## Implementation Notes

- Add session-level and run-level soul digest/profile references.
- Include refresh/reload semantics in CLI/HTTP/UDS.
- Avoid modifying task lease authority; this is provenance, not ownership.
- Keep `ClaimNextRun` purely transactional and free of file I/O; claim code copies preloaded session soul provenance only.
