# ADR-009: Separate Session Health From HEARTBEAT.md

## Status

Accepted

## Date

2026-05-01

## Context

"Heartbeat for a normal session" can mean active prompt activity, idle session presence, task-run lease renewal, or authored wake instructions. AGH already has active prompt supervision through `[session.supervision]`, task-run lease heartbeat through task APIs, and network greet presence for peers.

The missing runtime primitive is explicit idle session health/presence for deciding whether a session is alive, attachable, and eligible for synthetic wake.

## Decision

The Heartbeat feature separates four concepts:

- Task Run Lease Heartbeat: task ownership and lease renewal.
- Session Activity Heartbeat: active prompt progress and inactivity supervision.
- Session Presence / Health Heartbeat: idle/runtime health, attachability, and wake eligibility.
- Agent `HEARTBEAT.md` Wake Policy: authored reentry/wake guidance.

`HEARTBEAT.md` consumes session health/presence but does not implement or replace liveness. If the current runtime lacks sufficient idle health data, the implementation must add or reinforce metadata-only `session.health` / `session.liveness` before using the policy for wake decisions.

Wake eligibility and synthetic prompt enqueue must be serialized through the session-scoped prompt gate. If a normal prompt starts after health was read but before synthetic wake enqueue, the wake path skips with deterministic reason `session_prompt_active_race`.

## Alternatives Considered

### Alternative 1: Fold Session Health Into HEARTBEAT.md

- **Description**: Let `HEARTBEAT.md` define when a session is healthy or stale.
- **Pros**: One feature surface.
- **Cons**: Author-authored prompt policy would control runtime liveness.
- **Why rejected**: Liveness must be daemon-owned runtime state.

### Alternative 2: Reuse Network Greet Presence

- **Description**: Treat AGH Network greet as session health.
- **Pros**: Existing heartbeat-like primitive.
- **Cons**: Peer presence is not local session attachability or wake eligibility.
- **Why rejected**: It would promote network presence beyond its authority.

## Consequences

### Positive

- Scheduler and manual wake decisions can use explicit health state.
- UI/CLI/API can explain why a session is or is not wake-eligible.
- `HEARTBEAT.md` remains prompt/policy, not liveness.

### Negative

- Adds runtime state and API work beyond the authored artifact.

### Risks

- Poor naming could reintroduce ambiguity. Avoid `agh session heartbeat`; use `status`, `health`, and `inspect`.

## Implementation Notes

- Example health payload fields: `state`, `health`, `last_activity_at`, `last_presence_at`, `active_prompt`, `eligible_for_wake`, and `ineligibility_reason`.
- Health updates are metadata-only and must not call the model, inject prompts, renew leases, or create task runs.
- `session_health` is a standalone table keyed by `session_id` and deleted through `sessions(id) ON DELETE CASCADE`.
- On daemon restart, health is recomputed from runtime/session state before any wake decision; stale persisted health alone cannot make a session wake-eligible.
- `session.health.update.after` fires only on state, health, or eligibility transitions and is capped by `[agents.heartbeat].session_health_hook_min_interval`.
