# ADR-001: Treat SOUL.md as an Optional Persona Artifact

## Status

Accepted

## Date

2026-05-01

## Context

AGH already has a structured agent definition in `AGENT.md`, capability catalogs, runtime context assembly, task leases, session supervision, and AGH Network presence. Reference systems use `SOUL.md` as a persona/identity artifact, but they separate it from runtime liveness and task ownership. The AGH design must preserve existing runtime authorities and avoid a second operational state model hidden in Markdown.

## Decision

AGH will support an optional `SOUL.md` file beside each agent's `AGENT.md`. `SOUL.md` is an authored persona and behavioral identity artifact. It uses Markdown with optional strict frontmatter plus a narrative body. It may define only persona/identity behavior such as role, communication style, decision principles, behavioral constraints, collaboration style, and memory policy hints.

`SOUL.md` must not define fields owned by `AGENT.md`, config, capabilities, task/runtime state, scheduler behavior, lease heartbeat, session activity, or AGH Network presence. Unknown frontmatter fields and forbidden operational fields fail validation.

If `SOUL.md` is absent, the agent remains valid with `present=false`. If `SOUL.md` exists but is malformed, oversized beyond the hard parser limit, outside the allowed path, or declares forbidden operational authority, agent profile loading fails closed and session creation must not proceed. CLI, HTTP, and UDS surfaces return the same typed diagnostic shape for the same failure class.

## Alternatives Considered

### Alternative 1: Extend AGENT.md Only

- **Description**: Add persona fields directly to `AGENT.md` and do not introduce `SOUL.md`.
- **Pros**: One canonical file; fewer discovery paths.
- **Cons**: Mixes operational agent definition with narrative persona; weakens authoring ergonomics.
- **Why rejected**: The user wants `SOUL.md` as an optional additional file, and competitor evidence shows a separate persona file improves authoring clarity.

### Alternative 2: Make SOUL.md the Agent Definition

- **Description**: Let `SOUL.md` replace or override agent identity, tools, routing, and runtime fields.
- **Pros**: Simple mental model for authors.
- **Cons**: Conflicts with existing AGH structured surfaces and could bypass validation.
- **Why rejected**: `SOUL.md` is persona, not provider/tool/runtime authority.

## Consequences

### Positive

- Keeps persona authoring clear and approachable.
- Preserves AGH's existing agent/runtime contracts.
- Allows strict validation without making the narrative body overly rigid.

### Negative

- Adds another optional file to the agent directory.
- Requires clear docs for precedence and allowed frontmatter fields.

### Risks

- Authors may try to place operational instructions in `SOUL.md`; strict validation and documentation must reject that.

## Implementation Notes

- Add a `SoulDef` / resolved soul type near existing agent configuration loading.
- Reuse the existing frontmatter parser style where possible.
- Keep `AGENT.md` mandatory and canonical for provider, command, tools, hooks, and prompt base.
- Do not implement silent fallback to "last good" soul in v1; invalid authored persona files are explicit authoring errors.
