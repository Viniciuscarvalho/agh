# ADR-006: Provide Managed Soul Authoring In V1

## Status

Accepted

## Date

2026-05-01

## Context

AGH features are incomplete when agents and extensions can only observe them but cannot operate them through structured runtime surfaces. The initial Agent Soul design exposed resolved soul metadata through prompt, context, CLI/API/UDS, hooks, and Host API, but kept extension mutation out of v1.

That read-only posture protected authority boundaries, but it did not satisfy AGH's extensible-and-agent-manageable product premise. `SOUL.md` is an authored agent identity file. If AGH can validate and resolve it, AGH should also provide the safe path to create, update, delete, inspect history, and roll it back.

The risk is not mutation itself. The risk is unmanaged mutation: direct file writes from several surfaces, hidden prompt overlays, hook-driven prompt injection, or extension-specific behavior that bypasses validation and session/task lifecycle guarantees.

## Decision

Agent Soul v1 includes a managed authoring plane. CLI, HTTP, UDS, Host API, and future web UI surfaces must all call one `SoulAuthoringService` that owns validation, compare-and-swap digest checks, atomic file writes, delete, revision history, rollback, re-resolution, and audit events.

The managed surfaces are:

- `agh agent soul write`
- `agh agent soul delete`
- `agh agent soul history`
- `agh agent soul rollback`
- HTTP/UDS equivalents for put/delete/history/rollback
- Host API actions `agents/soul/put`, `agents/soul/delete`, `agents/soul/history`, and `agents/soul/rollback`

Read, validate, and inspect surfaces remain available:

- `agh agent soul validate`
- `agh agent soul inspect`
- `agents/soul/get`
- `agents/soul/validate`

Write/delete/rollback require exact action grants for extensions. Hooks, bundles, tools/resources, MCP sidecars, bridge adapters, and network peers cannot mutate soul directly. Active sessions and active task runs never change implicitly after authoring mutations; explicit idle-session refresh remains required.

## Alternatives Considered

### Alternative 1: Read-Only V1

- **Description**: Ship prompt inclusion, read models, hook provenance, and Host API read access only.
- **Pros**: Smaller implementation scope and lower immediate write-safety risk.
- **Cons**: Leaves `SOUL.md` authoring as out-of-band filesystem work and weakens agent manageability.
- **Why rejected**: AGH's runtime premise requires structured agent-operable management, not just observability.

### Alternative 2: Direct File Mutation By Each Surface

- **Description**: CLI, HTTP, UDS, Host API, and web each write `SOUL.md` with local validation.
- **Pros**: Simple to wire per transport.
- **Cons**: Duplicates validation, creates inconsistent conflict handling, and makes audit/revision guarantees brittle.
- **Why rejected**: AGH requires one authority for mutation semantics and deterministic errors.

### Alternative 3: Extension-Owned Persona Overlays

- **Description**: Let extensions, bundles, or resources attach persona overlays that merge into prompt context.
- **Pros**: Highly flexible and powerful for external ecosystems.
- **Cons**: Reopens authority confusion with `AGENT.md`, capabilities, hooks, and runtime state; needs precedence and conflict semantics not proven yet.
- **Why rejected**: Overlay systems are future work. V1 must first establish the canonical managed authoring path for `SOUL.md`.

## Consequences

### Positive

- Makes Agent Soul fully agent-manageable in v1.
- Gives extensions a safe write path without prompt mutation backdoors.
- Provides audit history and rollback for persona changes.
- Keeps active session/task semantics deterministic.

### Negative

- Expands MVP scope with additional DTOs, routes, CLI verbs, Host API methods, storage, and tests.
- Requires careful atomic file write and CAS conflict handling.
- Requires SDK and docs updates beyond read-only exposure.

### Risks

- Stale writers could overwrite newer persona changes unless every write/delete/rollback enforces `expected_digest`.
- Rollback could restore content that is invalid under newer rules unless rollback revalidates before writing.
- Extension write grants could be too broad unless read and write actions are separate and exact.

## Implementation Notes

- Add `agent_soul_revisions` to the same numbered global DB migration as `agent_soul_snapshots`.
- Treat `expected_digest` as mandatory when replacing or deleting an existing `SOUL.md`.
- Persist successful put/delete/rollback revisions before returning success.
- Never log soul body content; revision body is durable authoring state, not observability payload.
- Keep hook payloads provenance-only.
- Keep active session/task refresh explicit and conflict-protected.
