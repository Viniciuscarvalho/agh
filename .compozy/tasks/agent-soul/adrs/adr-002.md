# ADR-002: Expose Resolved Soul Through Prompt and Read Models

## Status

Accepted

## Date

2026-05-01

## Context

Competitor evidence shows three patterns. OpenClaw injects `SOUL.md` as bounded prompt context and uses separate narrow structured identity. Hermes loads, scans, truncates, and places `SOUL.md` in the first system-prompt slot. Paperclip keeps `SOUL.md` as a sibling file referenced by `AGENTS.md`, which risks the file existing without actually being in prompt context.

AGH must ensure the model receives the resolved persona and that agents/operators can inspect what was applied through stable surfaces.

## Decision

AGH will resolve valid `SOUL.md` content directly and deterministically into the agent prompt path. AGH will also expose:

- a compact `soul` projection in `/agent/context`; and
- a complete resolved soul/profile read model through a separate endpoint or equivalent agent-operable surface.

The compact context projection will include only fields needed for live agent self-awareness and provenance. The full read model will include source path, digest, parsed frontmatter, body, truncation state, validation status, and provenance.

HTTP, UDS, and CLI surfaces will share the same `internal/api/contract` DTOs and `internal/api/core` behavior. UDS parity is part of the feature, not follow-up work: `agent.context.get`, `agent.soul.inspect`, `agent.soul.inspectDefinition`, `agent.soul.validate`, and `session.soul.refresh` must ship with the same request/response shapes as HTTP and CLI projections.

Extensions may read resolved soul through Host API method `agents/soul/get` when explicitly granted through the existing Host API allowlist. Hook payload contexts that already expose session/task/spawn state will include `SoulSnapshotID`, `SoulDigest`, and spawn-only `ParentSoulDigest` provenance fields; they do not expose soul body content.

The v1 extensibility contract is managed-authoring-first. Extension manifests, Host API methods, hook payloads, SDK constants, and protocol docs change so external code can discover, validate, write, delete, list history, roll back, and audit resolved soul metadata through explicit actions. Skills, capabilities, tools/resources, bundles, registries, bridge SDKs, MCP sidecars, and AGH Network protocol surfaces cannot mutate soul or inject persona content through side channels; they can only call managed Host API actions when explicitly authorized.

## Alternatives Considered

### Alternative 1: Prompt Only

- **Description**: Inject `SOUL.md` into prompt but expose no structured read model.
- **Pros**: Minimal API surface.
- **Cons**: Poor observability and weak agent-manageability.
- **Why rejected**: AGH requires agent-operable surfaces, not hidden prompt behavior only.

### Alternative 2: Read Model Only

- **Description**: Expose soul through `/agent/context` or API but do not inject it directly.
- **Pros**: Clean structured contract.
- **Cons**: The model may not receive persona unless it asks for context.
- **Why rejected**: Persona must be available at prompt construction.

### Alternative 3: Sibling Reference Only

- **Description**: Inject `AGENT.md` and tell the model to read `SOUL.md`.
- **Pros**: Mirrors Paperclip-style instruction bundles.
- **Cons**: Ambiguous, adapter-dependent, and not deterministic.
- **Why rejected**: AGH should guarantee applied persona context when `SOUL.md` is valid.

## Consequences

### Positive

- Ensures deterministic persona delivery.
- Makes resolved soul inspectable by agents and operators.
- Keeps dynamic runtime context separate from stable persona.
- Gives extensions a typed managed path without creating prompt-mutation backdoors.

### Negative

- Requires contract additions and OpenAPI/codegen updates.
- Adds payload budget considerations to `/agent/context`.
- Requires SDK/protocol documentation updates beyond the core HTTP/CLI surfaces.

### Risks

- The `soul` projection can bloat context if it includes full prose; keep `/agent/context` compact and move full body to the detailed read model.

## Implementation Notes

- Add `AgentSoulPayload` and `AgentSoulSectionPayload` to contract DTOs.
- Add typed request DTOs for soul validation, put, delete, history, rollback, and session soul refresh, and use them across HTTP, UDS, CLI, and Host API conversions.
- Render `soul` before runtime/task sections in situation context.
- Include diagnostics/provenance for truncation and digest.
- Keep source paths workspace-relative in payloads/events; do not expose full body in `/agent/context`, logs, metrics, task metadata, or observability events.
- Emit observability through existing `internal/logger` structured logs and `internal/observe` read surfaces where applicable; never log soul body content or absolute local paths.
- Add `agents/soul/get`, `agents/soul/validate`, `agents/soul/put`, `agents/soul/delete`, `agents/soul/history`, and `agents/soul/rollback` to `internal/extension/protocol/host_api.go`, `internal/extension/contract/host_api.go`, `internal/extension/host_api.go`, `sdk/go/host_api.go`, and regenerated TypeScript SDK contracts.
- Add tests proving Host API access requires explicit per-action grants and that hooks/resources/bundles/tools/MCP sidecars cannot bypass managed authoring.
