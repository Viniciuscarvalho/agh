# ADR-008: Persist Heartbeat Policy Snapshots and Wake Audit Without a Queue

## Status

Accepted

## Date

2026-05-01

## Context

AGH needs deterministic, agent-manageable `HEARTBEAT.md` behavior. OpenClaw keeps heartbeat schedule and coalescing mostly in memory, while Hermes stores durable run heartbeat on task rows and Paperclip uses a dual wake/run queue. None of those models maps directly to AGH.

AGH must preserve one durable work queue: `task_runs`.

## Decision

Persist `HEARTBEAT.md` through immutable snapshots and managed revisions:

- `agent_heartbeat_snapshots` for resolved policy, digest, diagnostics, projection, and source metadata.
- `agent_heartbeat_revisions` for managed authoring, CAS, history, delete, and rollback.
- A standalone `session_health` table for metadata-only health, attachability, and wake eligibility.
- Lightweight `agent_heartbeat_wake_state` and `agent_heartbeat_wake_events` side tables for wake attempts, coalescing, skip reasons, and diagnostics.

The wake state/audit tables are not queues. They contain no claimable work, no ownership, no lease, and no task status.

## Alternatives Considered

### Alternative 1: In-Memory Wake State Only

- **Description**: Copy OpenClaw-style in-memory coalescing and rely on logs.
- **Pros**: Simple and low overhead.
- **Cons**: Poor restart behavior, weak auditability, and limited agent inspection.
- **Why rejected**: AGH requires agent-manageable observability.

### Alternative 2: Store Everything in JSON Metadata

- **Description**: Put policy/wake state in session or task metadata JSON.
- **Pros**: Fewer migrations.
- **Cons**: Weak queryability and easy confusion with ownership state.
- **Why rejected**: Existing AGH lessons require side tables for queryable state and forbid dynamic ownership in JSON metadata.

### Alternative 3: Paperclip-Style Wake/Run Queue

- **Description**: Add durable `agent_wakeup_requests` and `heartbeat_runs`.
- **Pros**: Strong execution audit model.
- **Cons**: Creates a second durable queue.
- **Why rejected**: Conflicts with `task_runs` single-queue invariants.

## Consequences

### Positive

- Supports managed authoring, CAS, rollback, and audit.
- Keeps wake diagnostics inspectable after restart.
- Avoids a second work queue.

### Negative

- Adds schema and migration work.
- Requires clear table naming to prevent future queue semantics.

### Risks

- Future contributors may try to evolve wake events into claimable work. Tests and invariants must prevent that.

## Implementation Notes

- Agent Soul owns global DB migration v12. Heartbeat reserves global DB migration v13. If implementation order changes, the specs and ADRs must be updated together before code is written; do not silently reuse v12.
- Migration v13 creates standalone tables: `agent_heartbeat_snapshots`, `agent_heartbeat_revisions`, `session_health`, `agent_heartbeat_wake_state`, and `agent_heartbeat_wake_events`.
- `session_health.session_id` references `sessions(id) ON DELETE CASCADE`.
- `agent_heartbeat_wake_state.session_id` references `sessions(id) ON DELETE CASCADE`.
- `agent_heartbeat_wake_events.session_id` references `sessions(id) ON DELETE SET NULL`.
- `policy_snapshot_id` on wake tables references `agent_heartbeat_snapshots(id) ON DELETE SET NULL`.
- Wake audit rows must have explicit `result`, closed-enum `reason`, `session_id`, `policy_snapshot_id`, `created_at`, and `expires_at`.
- Wake events are retained for `[agents.heartbeat].wake_event_retention`, default 168 hours, then removed by bounded cleanup at daemon startup and during the scheduler maintenance cycle.
- Snapshot rollback creates a new revision and never mutates historical snapshot or revision rows.
- No raw task claim token or provider secret may be stored.
