# ADR-011: Make Config the Authority for Cadence and Wake Limits

## Status

Accepted

## Date

2026-05-01

## Context

OpenClaw supports heartbeat cadence, target, light context, isolated sessions, and active hours. Those controls are useful, but AGH already has separate config authorities for session supervision, scheduler behavior, and network greet intervals. Allowing `HEARTBEAT.md` to directly define daemon timing would make a prompt artifact a scheduler.

## Decision

`[agents.heartbeat]` config is the authority for cadence, active-hours bounds, wake cooldown, rate limits, body/projection sizes, and active-session-only behavior.

`HEARTBEAT.md` may express preferences inside those bounds, such as quiet-window preference, wake summary, prompt/reentry guidance, and context projection hints. It cannot redefine `[session.supervision]`, scheduler intervals, task lease duration, or `[network] greet_interval`.

Policy digests include the resolved config subset so config changes that affect heartbeat semantics cannot silently reuse stale policy snapshots. The digest algorithm is:

```text
sha256("agh.heartbeat.v1\n" || canonical_json(frontmatter) || "\n" || normalized_body || "\n" || canonical_json(resolved_config_subset))
```

Active-hours and quiet-window preferences are evaluated at wake-decision time. `active_hours` are allow windows with union semantics; `quiet_windows` are deny windows with union semantics and subtract from active hours. Timezones must be IANA names. Windows use local wall-clock `HH:MM`, and cross-midnight windows are allowed when `end <= start`.

## Alternatives Considered

### Alternative 1: Put Cadence and Delivery Fully in HEARTBEAT.md

- **Description**: Let each file define cadence, active hours, and delivery behavior.
- **Pros**: Flexible and author-friendly.
- **Cons**: Turns Markdown into scheduler authority.
- **Why rejected**: Conflicts with AGH config lifecycle and runtime authority boundaries.

### Alternative 2: No Cadence or Active-Hours in MVP

- **Description**: Use `HEARTBEAT.md` only as checklist/reentry guidance.
- **Pros**: Simpler.
- **Cons**: Loses a major proven control from OpenClaw.
- **Why rejected**: Config-owned bounds allow useful controls without giving authority to Markdown.

## Consequences

### Positive

- Keeps timing/rate control centralized and testable.
- Allows useful author preferences without undermining runtime safety.
- Provides clear config lifecycle and docs surface.

### Negative

- Authors cannot make arbitrary per-file timing decisions.

### Risks

- Weak validation could let file preferences exceed config bounds. Parser and resolver tests must cover this.

## Implementation Notes

- Add `[agents.heartbeat]` keys for `enabled`, `max_body_bytes`, `context_projection_bytes`, `min_interval`, `default_interval`, `wake_cooldown`, `max_wakes_per_cycle`, and `active_session_only`.
- Add `[agents.heartbeat]` keys for `wake_event_retention`, `session_health_stale_after`, and `session_health_hook_min_interval`.
- Prefer `active_session_only = true` in MVP.
- File preference fields must resolve to clamped, diagnostic-rich values.
- If `allow_active_hours_preferences=false`, file time-window preferences are ignored with `heartbeat_preference_ignored`.
- If preferences exceed config bounds, the resolver clamps them and emits `heartbeat_preference_clamped`.
- Invalid timezone names or malformed windows fail validation with deterministic diagnostics.
