# Analysis: acpx

## Scope

- [Direct] Inspected only `.resources/acpx/` and the repo-local instruction file `.resources/acpx/AGENTS.md`. The local instructions frame `acpx` as "the smallest useful ACP client" and warn that data models, config keys, flags, output shapes, and naming conventions are product surface (`.resources/acpx/AGENTS.md:27-46`).
- [Direct] Focused on ACP protocol/session state, built-in agent registry, conformance suite, lifecycle/progress/heartbeat/liveness, and the boundary between ACP wire state and runtime policy/state (`.resources/acpx/conformance/spec/v1.md:15-61`, `.resources/acpx/docs/2026-02-27-acpx-session-model.md:6-20`, `.resources/acpx/docs/2026-02-25-warm-session-owner-architecture.md:123-139`).
- [Direct] No files were written or modified by the subagent.

## Overview

- [Direct] `acpx` is a headless ACP client/backend rather than a full orchestration platform. It exists to let agents communicate over ACP without PTY scraping while providing session persistence, queueing, lifecycle handling, and harness-specific smoothing (`.resources/acpx/VISION.md:1-8`, `.resources/acpx/VISION.md:18-27`, `.resources/acpx/VISION.md:108-122`).
- [Direct] ACP conformance profile v1 is intentionally narrow: `initialize`, `session/new`, `session/prompt`, `session/update`, `session/cancel`, and baseline errors only. `session/load`, `session/fork`, `session/list`, `session/resume`, `session/set_model`, and `$/cancel_request` are out of scope or unstable (`.resources/acpx/conformance/spec/v1.md:15-61`, `.resources/acpx/conformance/spec/v1.md:84-91`, `.resources/acpx/docs/2026-02-19-acp-coverage-roadmap.md:57-73`).
- [Direct] The central design boundary is explicit: ACP stream data is raw JSON-RPC, while queue ownership, process health, locks, retries, offsets, diagnostics, and lifecycle snapshots are local runtime/checkpoint/status state (`.resources/acpx/docs/2026-02-27-acpx-session-model.md:14-20`, `.resources/acpx/docs/2026-02-27-acpx-session-model.md:54-79`, `.resources/acpx/docs/2026-02-27-acpx-session-model.md:137-150`).
- [Inference] AGH should treat ACP `session/update` state as protocol transcript/progress, not as the authority for daemon scheduling, autonomy policy, liveness, retries, queue depth, or supervision. `acpx` repeatedly models those as out-of-band runtime state (`.resources/acpx/docs/2026-03-25-acpx-flows-architecture.md:474-480`, `.resources/acpx/docs/2026-03-25-acpx-flows-architecture.md:512-536`).

## Mechanisms / Patterns

- [Direct] Session identity is three-way, not one overloaded id: `acpx_record_id` is local storage identity, `acp_session_id` is the ACP wire/session id, and `agent_session_id` is optional provider/harness identity. Values may coincide, but semantics remain distinct (`.resources/acpx/docs/2026-02-27-acpx-session-model.md:22-34`, `.resources/acpx/docs/2026-02-23-session-identity-spec.md:36-56`, `.resources/acpx/src/types.ts:302-331`).
- [Direct] Session persistence uses an append-only raw ACP stream plus a derived checkpoint. Stream lines are JSON-RPC request/response/error/notification messages, and `session.json` may include local bookkeeping but is not a transport protocol (`.resources/acpx/docs/2026-02-27-acpx-session-model.md:35-53`, `.resources/acpx/docs/2026-02-27-acpx-session-model.md:90-135`, `.resources/acpx/src/session/persistence/serialize.ts:5-41`).
- [Direct] Prompt progress comes from ACP `session/update` notifications. The runtime maps update tags such as `agent_message_chunk`, `agent_thought_chunk`, `tool_call`, `tool_call_update`, `usage_update`, `available_commands_update`, `current_mode_update`, `config_option_update`, `session_info_update`, and `plan` into local UI/runtime events (`.resources/acpx/src/runtime/public/contract.ts:12-25`, `.resources/acpx/src/session/conversation-model.ts:600-652`, `.resources/acpx/src/runtime/public/events.ts:236-342`).
- [Direct] Terminal turn result is separated from live events: `startTurn(...).events` carries live events, while `AcpRuntimeTurn.result` carries terminal `completed` / `cancelled` / `failed` state; `runTurn` exists as a compatibility adapter (`.resources/acpx/src/runtime/public/contract.ts:81-148`, `.resources/acpx/src/runtime/public/contract.ts:150-158`).
- [Direct] Queue ownership is a local actor model: one owner per session, serialized prompt/control mutations, local IPC, queue depth, TTL, PID, socket path, `ownerGeneration`, and heartbeat timestamps (`.resources/acpx/docs/2026-02-25-warm-session-owner-architecture.md:92-157`, `.resources/acpx/src/cli/queue/lease-store.ts:9-35`, `.resources/acpx/src/cli/session/queue-owner-runtime.ts:66-96`).
- [Direct] Heartbeat/liveness is runtime-local: `heartbeatAt` goes stale after 15s, owner heartbeat refresh runs every 5s, and health probes validate lease, PID, socket reachability, generation, and queue depth (`.resources/acpx/src/cli/queue/lease-store.ts:5-8`, `.resources/acpx/src/cli/queue/lease-store.ts:77-83`, `.resources/acpx/src/cli/session/queue-owner-runtime.ts:43-45`, `.resources/acpx/src/cli/session/queue-owner-runtime.ts:211-218`, `.resources/acpx/src/cli/queue/ipc-health.ts:4-63`).
- [Direct] Cancellation is cooperative protocol control, not process teardown: `session/cancel` is sent for active prompts, expected terminal result is `stopReason = "cancelled"`, and session state remains reusable for follow-up prompts (`.resources/acpx/src/acp/client.ts:832-860`, `.resources/acpx/docs/ACPX_ERROR_STRATEGY.md:101-107`, `.resources/acpx/docs/2026-02-19-acp-coverage-roadmap.md:35-39`).
- [Direct] Persistent session recovery is strict where context matters: persistent flow sessions must reconnect and `session/load` the same underlying session; silently replacing with a fresh session is forbidden when persistence semantics are required (`.resources/acpx/AGENTS.md:164-168`, `.resources/acpx/docs/2026-03-25-acpx-flows-architecture.md:435-456`, `.resources/acpx/src/runtime/engine/reconnect.ts:237-325`).
- [Direct] Agent definitions are a registry plus escape hatch: built-ins map names to ACP commands, default agent is `codex`, aliases map `factory-droid`/`factorydroid` to `droid`, unknown names pass through as raw commands, and `--agent` bypasses positional resolution (`.resources/acpx/src/agent-registry.ts:38-107`, `.resources/acpx/docs/2026-02-17-agent-registry.md:8-49`, `.resources/acpx/agents/README.md:1-32`).

## Transferable Patterns

- [Direct] Keep protocol transcript pure: append raw ACP JSON-RPC messages, derive checkpoints/projections separately, and reject custom stream envelopes (`.resources/acpx/docs/2026-02-27-acpx-session-model.md:54-79`, `.resources/acpx/docs/2026-02-27-acpx-session-model.md:179-192`). [Inference] AGH should persist ACP traffic as ACP, then project daemon/UI/autonomy state into separate tables/events.
- [Direct] Use explicit identity names for distinct layers (`acpxRecordId`, `acpSessionId`, `agentSessionId`) and avoid synthesizing missing provider ids (`.resources/acpx/docs/2026-02-23-session-identity-spec.md:50-88`, `.resources/acpx/README.md:327-331`). [Inference] AGH should similarly distinguish daemon session id, ACP wire session id, and provider/native session id.
- [Direct] Model liveness as runtime state, not ACP state: queue owner health is derived from lease + heartbeat + PID/socket checks; flow liveness uses `live.json` and `node_heartbeat`, not `session/update` (`.resources/acpx/docs/2026-02-25-warm-session-owner-architecture.md:134-139`, `.resources/acpx/docs/2026-03-26-acpx-flow-trace-replay.md:288-295`, `.resources/acpx/docs/2026-03-26-acpx-flow-trace-replay.md:567-584`). [Inference] AGH should expose daemon liveness/progress through AGH runtime events/status surfaces, while preserving ACP `session/update` for agent output.
- [Direct] Separate live stream from terminal result: `startTurn` returns live events and a result promise; compatibility terminal events are explicitly labeled as compatibility (`.resources/acpx/src/runtime/public/contract.ts:103-145`, `.resources/acpx/src/runtime/public/contract.ts:150-158`). [Inference] AGH should avoid using "latest protocol update" as the authoritative terminal state; terminal turn outcome should be a separate runtime fact.
- [Direct] Conformance is data-driven and case-based, with timeouts, `settle_timeout_ms`, background prompt/cancel, permission modes, update-count/session checks, and explicit error expectations (`.resources/acpx/conformance/README.md:75-96`, `.resources/acpx/conformance/cases/021-prompt-post-success-drain.json:1-50`, `.resources/acpx/conformance/runner/run.ts:979-1025`). [Inference] AGH can adopt the same fixture-style conformance harness for ACP adapter compatibility and late-update regressions.
- [Direct] Permission policy is resolved by the runtime/client layer (`approve-all`, `approve-reads`, `deny-all`, non-interactive behavior), while ACP carries permission requests/responses (`.resources/acpx/src/permissions.ts:98-152`, `.resources/acpx/src/filesystem.ts:87-167`, `.resources/acpx/docs/2026-03-28-acpx-flow-permission-requirements.md:40-103`). [Inference] AGH should keep autonomy/permission policy in runtime configuration and audit logs, not encode it as a fake ACP session state.

## Risks / Mismatches

- [Direct] The conformance README says the profile includes 20 required cases, but the JSON profile lists 21 required case ids (`.resources/acpx/conformance/README.md:33-36`, `.resources/acpx/conformance/profiles/acp-core-v1.json:6-28`). [Inference] Treat acpx conformance artifacts as useful but draft; AGH should verify case inventory before copying.
- [Direct] `session/load` is central to acpx crash reconnect and persistent-session behavior, but it is outside conformance v1 (`.resources/acpx/docs/2026-02-17-session-management.md:73-86`, `.resources/acpx/src/runtime/engine/reconnect.ts:282-325`, `.resources/acpx/conformance/spec/v1.md:84-91`). [Inference] AGH should not assume every ACP adapter supports load/resume unless capability-detected or covered by an AGH-specific profile.
- [Direct] `session/set_model` is labeled unstable/not yet supported in coverage docs, while runtime code has an `unstable_setSessionModel` path and persists desired/current model state (`.resources/acpx/docs/2026-02-19-acp-coverage-roadmap.md:57-73`, `.resources/acpx/src/acp/client.ts:797-829`, `.resources/acpx/src/types.ts:285-299`). [Inference] AGH should avoid promoting unstable ACP controls to hard runtime invariants without fallback/error semantics.
- [Direct] Runtime event parsing accepts derived/local shapes such as `client_operation` and `update` in addition to ACP `session/update`, while the session model forbids custom envelopes in the ACP stream (`.resources/acpx/src/runtime/public/events.ts:236-342`, `.resources/acpx/docs/2026-02-27-acpx-session-model.md:73-79`). [Inference] AGH must be precise about which streams are protocol transcripts versus rendered/runtime projections.
- [Direct] Queue ownership is local-host scoped and PID/socket based; cross-machine distributed queue ownership is a non-goal (`.resources/acpx/docs/2026-02-25-warm-session-owner-architecture.md:86-90`, `.resources/acpx/docs/CLI.md:344-357`). [Inference] AGH daemon semantics may need stronger distributed/process supervision than acpx's local CLI owner model.
- [Direct] Conformance runner permission modes are only `approve-all` and `deny-all`, while runtime permissions include `approve-reads` and non-interactive policy (`.resources/acpx/conformance/runner/run.ts:25-35`, `.resources/acpx/src/types.ts:28-35`, `.resources/acpx/src/permissions.ts:102-152`). [Inference] AGH permission conformance should test read-only and non-interactive denial/error behavior separately.
