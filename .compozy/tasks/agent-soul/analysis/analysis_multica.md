## Overview

- [Direct] Multica frames agents as first-class teammates: issues can be assigned to agents, agents comment/change status/report blockers, and agents appear on the board/activity surfaces (`.resources/multica/README.md:31`, `.resources/multica/README.md:45`, `.resources/multica/apps/docs/content/docs/agents.mdx`).
- [Direct] The core orchestration split is server-owned coordination plus local daemon execution: the server owns workspaces, issues, members, task queue, and realtime events; the daemon detects tools, registers runtimes, polls/heartbeats, creates isolated workspaces, and invokes AI CLIs (`.resources/multica/apps/docs/content/docs/how-multica-works.mdx`, `.resources/multica/CLI_AND_DAEMON.md`, `.resources/multica/server/internal/daemon/daemon.go`).
- [Direct] Work starts from four trigger families: assignment, `@mention`, chat, and scheduled/manual autopilot (`.resources/multica/apps/docs/content/docs/how-multica-works.mdx`, `.resources/multica/apps/docs/content/docs/tasks.mdx`, `.resources/multica/apps/docs/content/docs/autopilots.mdx`).
- [Direct] Multica has its own queue and lifecycle: `queued -> dispatched -> running -> completed/failed/cancelled`, represented by `agent_task_queue` and manipulated by SQL/service handlers (`.resources/multica/server/migrations/001_init.up.sql`, `.resources/multica/server/pkg/db/queries/agent.sql:67`, `.resources/multica/server/pkg/db/queries/agent.sql:164`, `.resources/multica/server/internal/service/task.go:393`).
- [Inference] For AGH, the useful idea is not Multica's queue table itself. It is the surrounding coordination model: role-aware actors, trigger normalization, best-effort wakeups, runtime liveness, transcript persistence, and status reconciliation, all routed through AGH's existing task lease authority.

## Mechanisms / Patterns

- [Direct] Agent/member role model. Multica stores agents separately from human members but lets both act in issues/comments/activity. Schema fields include `issue.assignee_type`, `issue.creator_type`, `comment.author_type`, `agent.visibility`, `agent.runtime_mode`, `agent.status`, and `agent.max_concurrent_tasks` (`.resources/multica/server/migrations/001_init.up.sql`, `.resources/multica/server/migrations/023_agent_concurrency_default.up.sql`, `.resources/multica/apps/docs/content/docs/agents.mdx`).
- [Direct] Assignment as orchestration trigger. Assigning a ready agent to a non-backlog issue enqueues work; reassignment/unassignment cancels active work and can enqueue for the new agent (`.resources/multica/apps/docs/content/docs/assigning-issues.mdx`, `.resources/multica/server/internal/handler/issue.go`, `.resources/multica/server/internal/service/task.go:101`).
- [Direct] Comment and mention collaboration. Member comments can trigger the assigned agent; explicit `@agent` mentions enqueue mention tasks; self-mentions are skipped, `@all` does not trigger agents, and thread/reply heuristics suppress noisy loops (`.resources/multica/apps/docs/content/docs/mentioning-agents.mdx`, `.resources/multica/server/internal/handler/comment.go`).
- [Direct] Queue claiming and serialization. `ClaimAgentTask` atomically moves `queued` to `dispatched` with `FOR UPDATE SKIP LOCKED`; code also checks per-agent max concurrency before claiming (`.resources/multica/server/pkg/db/queries/agent.sql:164`, `.resources/multica/server/internal/service/task.go:393`).
- [Direct] Daemon runtime loop. The daemon registers runtime capabilities, polls runtime IDs round-robin, claims tasks via HTTP, uses a daemon-level semaphore, starts tasks, streams messages, tracks cancellation, and completes/fails tasks (`.resources/multica/server/internal/daemon/daemon.go`, `.resources/multica/server/internal/daemon/client.go`).
- [Direct] Wakeups are hints, not correctness. WebSocket task-available notifications wake local daemons, but correctness remains in HTTP claim/DB transitions (`.resources/multica/server/internal/daemonws/hub.go`, `.resources/multica/server/internal/daemonws/notifier.go`, `.resources/multica/server/internal/daemon/wakeup.go`, `.resources/multica/server/internal/service/task.go:1198`).
- [Direct] Heartbeat/liveness. Runtime registration and heartbeat maintain `agent_runtime.status`/`last_seen_at`; stale runtimes are marked offline after 45s and offline runtimes expire after 7 days; stale dispatched/running tasks fail via sweeper paths (`.resources/multica/apps/docs/content/docs/daemon-runtimes.mdx`, `.resources/multica/server/pkg/db/queries/runtime.sql`, `.resources/multica/server/cmd/server/runtime_sweeper.go`).
- [Direct] Recovery and retry. Daemon startup calls recover-orphans for each runtime; orphaned dispatched/running tasks fail with `runtime_recovery`; retryable reasons include `runtime_offline`, `runtime_recovery`, and `timeout`; autopilot tasks do not auto-retry (`.resources/multica/server/internal/handler/task_lifecycle.go`, `.resources/multica/server/internal/service/task.go:851`, `.resources/multica/server/pkg/db/queries/agent.sql:268`).
- [Direct] Persistence of run context. Tasks persist attempts, parent task IDs, failure reasons, trigger summaries, trigger comments, task messages, session IDs, work dirs, usage, and elapsed time (`.resources/multica/server/migrations/026_task_messages.up.sql`, `.resources/multica/server/migrations/055_task_lease_and_retry.up.sql`, `.resources/multica/server/migrations/061_task_trigger_summary.up.sql`, `.resources/multica/server/pkg/db/queries/task_message.sql`, `.resources/multica/server/pkg/db/queries/agent.sql:214`).
- [Direct] Realtime status composition. Frontend presence derives availability from runtime reachability and workload from active tasks, then invalidates via task lifecycle events rather than polling every detail (`.resources/multica/packages/core/agents/types.ts`, `.resources/multica/packages/core/agents/derive-presence.ts`, `.resources/multica/packages/core/agents/queries.ts`, `.resources/multica/packages/core/realtime/use-realtime-sync.ts`).

## Transferable Patterns

- [Direct] Multica normalizes assignment, mention, chat, and autopilot into task-trigger metadata (`.resources/multica/apps/docs/content/docs/how-multica-works.mdx`, `.resources/multica/server/internal/service/task.go`, `.resources/multica/server/migrations/061_task_trigger_summary.up.sql`). [Inference] AGH can adopt this trigger taxonomy as lease-intent metadata while still creating/claiming work only through AGH task leases.
- [Direct] Multica treats agents as board participants with actor type, visibility, runtime binding, comments, activity, and assignment semantics (`.resources/multica/server/migrations/001_init.up.sql`, `.resources/multica/apps/docs/content/docs/agents.mdx`, `.resources/multica/server/internal/handler/comment.go`). [Inference] AGH can reuse the actor model for agent-manageable collaboration surfaces without coupling actor identity to a separate queue.
- [Direct] Wakeups are deliberately best-effort: daemon WebSocket notifications only prompt polling/claiming, while DB/HTTP claim paths remain authoritative (`.resources/multica/server/internal/daemonws/hub.go`, `.resources/multica/server/internal/daemon/wakeup.go`, `.resources/multica/server/internal/service/task.go:1198`). [Inference] This transfers cleanly if wakeups merely nudge AGH's existing lease claimant and never allocate work themselves.
- [Direct] Runtime liveness is separated from work status through `agent_runtime` heartbeat/stale sweeps and active task status reconciliation (`.resources/multica/server/pkg/db/queries/runtime.sql`, `.resources/multica/server/cmd/server/runtime_sweeper.go`, `.resources/multica/packages/core/agents/derive-presence.ts`). [Inference] AGH should keep runtime/session heartbeat as liveness evidence and let task leases remain the source of truth for ownership, expiry, and cancellation.
- [Direct] Claiming enforces serialization/concurrency near the state transition, not in the UI: max concurrent task checks happen in `TaskService.ClaimTask`, and SQL claim uses `FOR UPDATE SKIP LOCKED` (`.resources/multica/server/internal/service/task.go:393`, `.resources/multica/server/pkg/db/queries/agent.sql:164`). [Inference] Equivalent constraints should live in AGH's lease acquisition transaction, not in a parallel dispatcher.
- [Direct] Multica pins resumable execution context early and preserves it through failure/retry (`.resources/multica/server/pkg/db/queries/agent.sql:214`, `.resources/multica/server/pkg/db/queries/agent.sql:238`, `.resources/multica/server/internal/handler/task_lifecycle.go`, `.resources/multica/server/internal/daemon/daemon.go`). [Inference] Persisting session/workdir/provider context on the lease/run record would improve crash recovery without changing queue ownership.
- [Direct] Comment-triggered collaboration has explicit loop/noise guards: self-mention skip, `@all` non-trigger, mention-suppression for assignee, thread inheritance limits, and pending-task dedup (`.resources/multica/apps/docs/content/docs/mentioning-agents.mdx`, `.resources/multica/server/internal/handler/comment.go`, `.resources/multica/server/internal/handler/issue.go`). [Inference] These are strong candidates for AGH collaboration policy, especially if AGH exposes agent comments or cross-agent delegation.
- [Direct] Autopilot's safer path creates an issue and then uses the normal assignment/task flow; direct `run_only` exists but docs label it unstable/not fully wired (`.resources/multica/apps/docs/content/docs/autopilots.mdx`, `.resources/multica/CLI_AND_DAEMON.md`, `.resources/multica/server/internal/service/autopilot.go`). [Inference] Scheduled automation should create or attach to an AGH-managed task/lease rather than dispatching an independent run queue.
- [Direct] Task messages are persisted and broadcast separately from lifecycle state, with frontend avoiding invalidation storms for message events (`.resources/multica/server/migrations/026_task_messages.up.sql`, `.resources/multica/server/pkg/db/queries/task_message.sql`, `.resources/multica/packages/core/realtime/use-realtime-sync.ts`). [Inference] AGH can separate durable transcript streaming from lease lifecycle transitions.

## Risks / Mismatches

- [Direct] Multica's central orchestration primitive is `agent_task_queue` with claim/start/complete/fail endpoints (`.resources/multica/server/migrations/001_init.up.sql`, `.resources/multica/server/pkg/db/queries/agent.sql`, `.resources/multica/server/internal/handler/daemon.go`). [Inference] Porting this table or claim API directly would create a parallel queue and risk bypassing AGH task leases.
- [Direct] Autopilot has a direct `run_only` task path via `CreateAutopilotTask`, while docs/CLI state this mode is not fully supported or is unstable (`.resources/multica/server/pkg/db/queries/autopilot.sql`, `.resources/multica/server/internal/service/autopilot.go`, `.resources/multica/apps/docs/content/docs/autopilots.mdx`, `.resources/multica/CLI_AND_DAEMON.md`). [Inference] Avoid direct scheduled-run dispatch unless it is represented as a normal AGH lease-backed task.
- [Direct] Multica added task `last_heartbeat_at`, but stale-task handling still primarily uses dispatched/running timeout windows and runtime heartbeat paths (`.resources/multica/server/migrations/055_task_lease_and_retry.up.sql`, `.resources/multica/server/pkg/db/queries/agent.sql:268`, `.resources/multica/server/cmd/server/runtime_sweeper.go`). [Inference] Do not confuse runtime heartbeat or progress heartbeat with lease renewal; AGH lease expiry/renewal must remain explicit and authoritative.
- [Direct] There is possible schema/code/doc tension around pending task uniqueness and parallelism: docs/code describe same-agent dedup while allowing different agents on the same issue, but migration `022_task_lifecycle_guards` creates a partial unique index named `idx_one_pending_task_per_issue` (`.resources/multica/apps/docs/content/docs/assigning-issues.mdx`, `.resources/multica/apps/docs/content/docs/mentioning-agents.mdx`, `.resources/multica/server/pkg/db/queries/agent.sql:164`, `.resources/multica/server/migrations/022_task_lifecycle_guards.up.sql`). [Inference] Re-check the final effective schema before copying any "parallel agents per issue" invariant.
- [Direct] SQL comments for the agent task snapshot mention sticky failed-state visibility, but frontend presence derivation treats workload as active-task-only and says failures/history belong elsewhere (`.resources/multica/server/pkg/db/queries/agent.sql:385`, `.resources/multica/packages/core/agents/derive-presence.ts`). [Inference] Keep presence, failure history, and health status as separate dimensions to avoid misleading UI state.
- [Direct] Empty-claim caching and Redis wakeup relay are queue-specific optimizations built around runtime queue versions and daemon claim attempts (`.resources/multica/server/internal/service/empty_claim_cache.go`, `.resources/multica/server/internal/daemonws/notifier.go`, `.resources/multica/server/internal/service/task.go:458`). [Inference] These are transferable only as versioned negative-cache/nudge patterns around AGH's lease query, not as a second scheduler.
- [Direct] The event bus is synchronous in-process with panic isolation, and server listeners fan out state changes (`.resources/multica/server/internal/events/bus.go`, `.resources/multica/server/cmd/server/listeners.go`). [Inference] Fine for local modularity, but if AGH needs durable coordination across daemon/process boundaries, an in-process bus alone is not enough.
- [Direct] Mention docs acknowledge indirect agent mention cycles are possible; only direct self-mention loops are guarded (`.resources/multica/apps/docs/content/docs/mentioning-agents.mdx`, `.resources/multica/server/internal/handler/comment.go`). [Inference] Agent-to-agent collaboration needs stronger cycle/rate/depth controls if AGH exposes recursive delegation.

## Open Questions

- [Inference] What is AGH's canonical lease-backed object/API that should receive Multica-like triggers from assignment, mention, chat, and schedule? This must be mapped before adopting patterns from `.resources/multica/server/internal/service/task.go` or `.resources/multica/server/pkg/db/queries/agent.sql`.
- [Inference] Should AGH allow multiple agents to collaborate on the same user-visible task/issue concurrently, as Multica intends in docs (`.resources/multica/apps/docs/content/docs/assigning-issues.mdx`, `.resources/multica/apps/docs/content/docs/mentioning-agents.mdx`), or should AGH enforce one active lease per task with explicit subleases?
- [Inference] When a new comment arrives while an agent is already running, should AGH coalesce it into the active session, queue it as a follow-up lease intent, or ignore it until the lease completes? Multica mostly dedups pending work and suppresses noisy comments (`.resources/multica/server/internal/handler/comment.go`, `.resources/multica/server/internal/handler/issue.go`).
- [Inference] Does AGH need an autopilot equivalent that creates first-class tasks/issues, or only scheduled lease intents? Multica favors `create_issue` and labels `run_only` as unstable/not fully supported (`.resources/multica/apps/docs/content/docs/autopilots.mdx`, `.resources/multica/CLI_AND_DAEMON.md`).
- [Inference] Which AGH stream should own durable task messages and presence invalidation? Multica persists task messages separately and avoids invalidation storms (`.resources/multica/server/migrations/026_task_messages.up.sql`, `.resources/multica/packages/core/realtime/use-realtime-sync.ts`).
