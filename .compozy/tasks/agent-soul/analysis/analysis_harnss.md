# Analysis: harnss

## Scope

- [Direct] This review is limited to `.resources/harnss/` and local instruction files under that tree: `.resources/harnss/AGENTS.md`, `.resources/harnss/CLAUDE.md`, `.resources/harnss/README.md`, and `.resources/harnss/.agents/skills/vercel-react-best-practices/AGENTS.md`.
- [Direct] No source files were edited. The explored code paths are under `.resources/harnss/electron/src/...`, `.resources/harnss/src/...`, and `.resources/harnss/shared/...`.
- [Direct] The requested focus areas map to Harnss subsystems:

  - Agent/session orchestration: `.resources/harnss/electron/src/ipc/claude-sessions.ts`, `.resources/harnss/electron/src/ipc/acp-sessions.ts`, `.resources/harnss/electron/src/ipc/codex-sessions.ts`
  - Renderer orchestration: `.resources/harnss/src/hooks/useSessionManager.ts`, `.resources/harnss/src/hooks/session/useSessionPane.ts`, `.resources/harnss/src/hooks/session/useSessionLifecycle.ts`
  - Persistence: `.resources/harnss/electron/src/ipc/sessions.ts`, `.resources/harnss/shared/lib/session-persistence.ts`, `.resources/harnss/electron/src/lib/data-dir.ts`
  - Background tasks/agents: `.resources/harnss/src/lib/background/session-store.ts`, `.resources/harnss/src/lib/background/agent-store.ts`, `.resources/harnss/src/hooks/useBackgroundAgents.ts`

- [Inference] Harnss is most useful to AGH as a reference for multi-agent process/session orchestration, event normalization, task/tool correlation, and UI/runtime boundary discipline. It is less useful as a persistence or distributed-liveness model because Harnss is a local Electron app with in-memory process maps and JSON file session storage.

## Overview

- [Direct] `.resources/harnss/AGENTS.md` and `.resources/harnss/CLAUDE.md` describe Harnss as an Electron 40 + React 19 desktop app for the Agent Client Protocol, with multiple concurrent sessions, persistent chat history, project workspaces, background agents, permissions, and context compaction.
- [Direct] `.resources/harnss/README.md` positions Harnss as a cross-platform desktop application for running, managing, and switching between AI coding agents including Claude Code, Codex, and any ACP-compatible agent.
- [Direct] `.resources/harnss/package.json` includes `@agentclientprotocol/sdk`, `@anthropic-ai/claude-agent-sdk`, Electron, `node-pty`, Zustand, and MCP SDK dependencies. This matches the observed architecture: Electron main process owns subprocess/RPC work; React owns presentation and local interaction state.
- [Direct] The main process centralizes IPC registration in `.resources/harnss/electron/src/main.ts`: it imports and registers modules for spaces, projects, sessions, files, Claude sessions, terminal, git, agent registry, ACP sessions, Codex sessions, MCP, settings, and Jira.
- [Direct] `.resources/harnss/electron/src/main.ts` creates a hardened renderer boundary with `preload`, `contextIsolation: true`, and `nodeIntegration: false`, while `.resources/harnss/electron/src/preload.ts` exposes one `window.claude` bridge containing Claude, ACP, Codex, project, session, MCP, and utility methods.
- [Inference] Harnss' architectural center is not a single agent protocol. It is a harness that wraps several incompatible agent runtimes behind a shared renderer/session model. That shape is directly relevant to AGH's agents/tasks/network harness because AGH also needs to coordinate heterogeneous agents through stable runtime surfaces.

## Mechanisms / Patterns

- [Direct] Harnss separates process ownership from UI state. Electron main owns subprocesses and protocol clients in `.resources/harnss/electron/src/ipc/claude-sessions.ts`, `.resources/harnss/electron/src/ipc/acp-sessions.ts`, and `.resources/harnss/electron/src/ipc/codex-sessions.ts`; React hooks consume events and maintain render-facing state in `.resources/harnss/src/hooks/useClaude.ts`, `.resources/harnss/src/hooks/useACP.ts`, and `.resources/harnss/src/hooks/useCodex.ts`.
- [Direct] Claude sessions use an async input channel. `.resources/harnss/shared/lib/async-channel.ts` implements a push-based async iterable queue, and `.resources/harnss/electron/src/ipc/claude-sessions.ts` passes that channel into the Claude SDK `query` call, pushes user messages from IPC, and closes the channel on stop.
- [Direct] Claude event forwarding is stream-oriented. `.resources/harnss/electron/src/ipc/claude-sessions.ts` iterates the SDK query handle, attaches `_sessionId`, increments an event counter, forwards events with `safeSend("claude:event", ...)`, tracks permissions, and emits `claude:exit` on teardown.
- [Direct] ACP sessions are process-backed JSON/NDJSON connections. `.resources/harnss/electron/src/ipc/acp-sessions.ts` defines `ACPSessionEntry` with process, connection, ACP session ID, pending permissions, auth state, utility sessions, and stderr tracking. It spawns the agent command, initializes ACP client capabilities, handles session updates, permission requests, file read/write callbacks, auth, start, reload, cancel, and stop.
- [Direct] Codex sessions wrap `codex app-server` as JSON-RPC. `.resources/harnss/electron/src/ipc/codex-sessions.ts` starts the app server, initializes RPC, creates or resumes threads, forwards notifications to the renderer, bridges server-side approval requests, and tracks active turn IDs. `.resources/harnss/shared/lib/codex-rpc.ts` provides request IDs, pending request timeouts, notification dispatch, response dispatch, and process-exit cleanup.
- [Direct] Harnss normalizes incompatible event shapes into a shared UI model. `.resources/harnss/src/lib/engine/acp-adapter.ts` converts ACP tool calls/results into Claude-like tool/result structures, merges partial tool input from `tool_call` and `tool_call_update`, normalizes stdout/content, and derives tool names from ACP update kinds/titles. `.resources/harnss/src/lib/engine/acp-task-adapter.ts` detects Task/Agent-style tool calls and extracts task summary steps.
- [Direct] Renderer orchestration always mounts all engine hooks. `.resources/harnss/src/hooks/session/useSessionPane.ts` explicitly says it always calls `useClaude`, `useACP`, and `useCodex`, with dormant hooks when IDs are null, then selects the active engine shape for the pane.
- [Direct] Shared engine state lives in `.resources/harnss/src/hooks/useEngineBase.ts`, which owns message arrays, streaming state, processing state, cost/session stats, permission state, requestAnimationFrame scheduling, reset behavior, and cleanup. Engine-specific hooks layer protocol-specific event handling on top.
- [Direct] Session lifecycle has draft/eager materialization. `.resources/harnss/src/hooks/session/useDraftMaterialization.ts` pre-starts Claude/ACP sessions, prefetches Codex model state, probes MCP status, abandons unused draft sessions, and materializes draft sessions into persisted `ChatSession` records only when needed.
- [Direct] Message queueing is boundary-aware. `.resources/harnss/src/hooks/session/useMessageQueue.ts` queues messages per session, drains one message by active engine, supports background continuation, and can defer interruption until stream/tool boundaries.
- [Direct] Background task/session continuity is handled separately from visible UI state. `.resources/harnss/src/lib/background/session-store.ts` stores background session state by session ID, including messages, stats, processing flags, pending permissions, parent tool maps, current streaming IDs, Codex plan state, and active task metadata.
- [Direct] Background agent tracking is event-driven. `.resources/harnss/src/lib/background/agent-store.ts` tracks pending and confirmed background agents, task progress, tool progress, notification completion, and XML-tagged user messages. `.resources/harnss/src/hooks/useBackgroundAgents.ts` explicitly says background agents are event-driven and do not poll files.
- [Direct] Permissions use request IDs and pending resolver maps. Claude tracks `pendingPermissions` in `.resources/harnss/electron/src/ipc/claude-sessions.ts`; ACP tracks pending permission resolvers in `.resources/harnss/electron/src/ipc/acp-sessions.ts`; Codex forwards app-server approval requests in `.resources/harnss/electron/src/ipc/codex-sessions.ts` and maps them to renderer permission state in `.resources/harnss/src/hooks/useCodex.ts`.
- [Direct] Liveness is lifecycle/event based, not heartbeat based. `.resources/harnss/electron/src/ipc/acp-sessions.ts` uses process `error`/`exit` handlers and timeout wrappers; `.resources/harnss/shared/lib/codex-rpc.ts` exposes `isAlive` and rejects pending requests on process exit; `.resources/harnss/electron/src/ipc/claude-sessions.ts` sends `claude:exit` when the async SDK loop finishes.
- [Inference] Harnss' dominant pattern is "protocol-specific process supervisor + event adapter + shared session UI contract." This is highly transferable to AGH if implemented in the daemon/runtime layer rather than as renderer-owned orchestration.

## Transferable Patterns

- [Direct] The multi-engine structure in `.resources/harnss/src/hooks/session/useSessionPane.ts`, `.resources/harnss/src/hooks/useEngineBase.ts`, `.resources/harnss/src/hooks/useClaude.ts`, `.resources/harnss/src/hooks/useACP.ts`, and `.resources/harnss/src/hooks/useCodex.ts` gives Claude, ACP, and Codex a common renderer-facing shape. [Inference] AGH can reuse the pattern as a daemon-side "agent runtime adapter" contract: each provider/runtime owns spawn/connect/send/stop/permission semantics, then emits normalized session/task/tool events to CLI, HTTP/SSE, UDS, and web consumers.
- [Direct] `.resources/harnss/src/lib/engine/acp-adapter.ts` and `.resources/harnss/src/lib/engine/acp-task-adapter.ts` normalize ACP-specific events into shared tool/task shapes. [Inference] AGH should apply the same idea to its existing agents/tasks/network harness: keep provider-native details at the edge, then normalize into AGH event envelopes with stable IDs, parent IDs, task IDs, tool IDs, status, permissions, and result payloads.
- [Direct] Claude task/subagent routing in `.resources/harnss/src/hooks/useClaude.ts` uses parent tool maps and `parent_tool_use_id` to route subagent events into task cards. Background state persists those correlations in `.resources/harnss/src/lib/background/session-store.ts`. [Inference] Parent-child event correlation is directly relevant to AGH tasks and network coordination. AGH should preserve parent task/tool/session correlation IDs in durable event storage so UI, CLI, and network peers can reconstruct task trees without relying on renderer memory.
- [Direct] Codex tooling in `.resources/harnss/src/hooks/useCodex.ts` uses deterministic item-derived IDs such as `codex-tool-${item.id}` for tool calls and fallback result matching. [Inference] Deterministic event IDs are useful for AGH's network harness because replay, deduplication, SSE reconnection, CLI polling, and distributed task updates all benefit from stable IDs rather than UI-generated random IDs.
- [Direct] Permission bridges in `.resources/harnss/electron/src/ipc/claude-sessions.ts`, `.resources/harnss/electron/src/ipc/acp-sessions.ts`, and `.resources/harnss/electron/src/ipc/codex-sessions.ts` all converge on pending request IDs and later responses. [Inference] AGH can transfer this as a provider-neutral approval workflow: runtime emits `permission.requested`, stores a pending resolver/state row, and allows agents/operators to answer through CLI, REST, UDS, or web.
- [Direct] `.resources/harnss/src/hooks/session/useMessageQueue.ts` can queue messages and defer interruption until stream/tool boundaries. [Inference] AGH can adapt this as daemon-level turn control: avoid blindly interrupting providers mid-tool; model interruption as a lifecycle request that is honored at safe protocol boundaries when the provider supports it.
- [Direct] `.resources/harnss/electron/src/ipc/sessions.ts` uses `.meta.json` sidecars for fast listing and search fallback, while `.resources/harnss/shared/lib/session-persistence.ts` defines metadata fields such as engine, model, effort, permission mode, plan mode, Codex thread ID, folder, branch, and agent ID. [Inference] AGH should not copy JSON sidecars if SQLite/event stores already exist, but the indexed metadata set is valuable: session list views and CLI queries need cheap access to engine, agent, branch, folder, last user message, and last activity.
- [Direct] `.resources/harnss/src/lib/background/agent-store.ts` and `.resources/harnss/src/hooks/useBackgroundAgents.ts` implement event-driven background agent updates rather than polling. [Inference] This fits AGH better than polling task output files. AGH should drive background task/agent status from daemon events and durable task state, then fan out over SSE/UDS/network.

## Risks / Mismatches

- [Direct] Harnss is an Electron desktop app. `.resources/harnss/electron/src/main.ts` owns BrowserWindow creation and IPC registration, and `.resources/harnss/electron/src/preload.ts` exposes local IPC APIs to a React renderer. [Inference] AGH is a daemon-first runtime with CLI/HTTP/SSE/UDS and network concerns. Harnss' renderer-owned orchestration should not be copied directly; AGH should move equivalent orchestration into daemon services and expose it through agent-manageable surfaces.
- [Direct] Live sessions are held in process-local maps: Claude in `.resources/harnss/electron/src/ipc/claude-sessions.ts`, ACP in `.resources/harnss/electron/src/ipc/acp-sessions.ts`, and Codex in `.resources/harnss/electron/src/ipc/codex-sessions.ts`. [Inference] This is acceptable for a single desktop app process but insufficient as AGH's source of truth. AGH needs durable session/task state, restart recovery, and network-visible lifecycle semantics.
- [Direct] Harnss persistence uses JSON session files plus metadata sidecars in `.resources/harnss/electron/src/ipc/sessions.ts` and path helpers in `.resources/harnss/electron/src/lib/data-dir.ts`. The generic store in `.resources/harnss/electron/src/lib/json-file-store.ts` uses atomic temp+rename for non-sensitive JSON stores, but session saves in `.resources/harnss/electron/src/ipc/sessions.ts` write the main JSON file directly. [Inference] AGH should prefer its SQLite/event persistence model for sessions/tasks rather than adopting direct JSON session writes. If file artifacts are needed, they should be transactional or derived from durable state.
- [Direct] There is no dedicated heartbeat/liveness protocol in the explored Harnss runtime paths. Liveness is inferred from process exits, stream termination, request timeouts, and `CodexRpcClient.isAlive` in `.resources/harnss/shared/lib/codex-rpc.ts`. [Inference] That is too weak for AGH network coordination. Remote peers, daemon restarts, long-running tasks, and network partitions need explicit heartbeat, lease, deadline, or checkpoint semantics beyond local child-process exit events.
- [Direct] `.resources/harnss/CLAUDE.md` describes background agent polling behavior, but `.resources/harnss/src/hooks/useBackgroundAgents.ts` explicitly says background agents are event-driven and "No file polling needed." [Inference] Treat Harnss docs as useful orientation but verify implementation before transferring patterns. For AGH, prefer the implementation's event-driven model over stale polling-oriented documentation.
- [Direct] Harnss includes compatibility/migration-oriented local data behavior, such as the `openacpui-data` naming in `.resources/harnss/electron/src/lib/data-dir.ts` and plaintext migration behavior for sensitive stores in `.resources/harnss/electron/src/lib/json-file-store.ts`. [Inference] AGH's greenfield alpha posture should avoid copying compatibility bridges, legacy aliases, or migration fallbacks unless a current AGH schema migration explicitly requires them.
- [Direct] ACP file operations are exposed from main-process ACP callbacks in `.resources/harnss/electron/src/ipc/acp-sessions.ts` through `readTextFile` and `writeTextFile`. [Inference] AGH should be stricter about file/tool authority boundaries. The pattern is useful, but AGH should route file writes through its permission, audit, sandbox, and agent-manageability model rather than simple callback exposure.

## Open Questions

- [Direct] `.resources/harnss/electron/src/ipc/acp-sessions.ts` supports `loadSession` revival for ACP sessions when the agent advertises support, while `.resources/harnss/electron/src/ipc/codex-sessions.ts` supports `thread/resume`. Open question: Which AGH providers support true resumable provider sessions versus AGH-only replay/reconstruction from stored events?
- [Direct] `.resources/harnss/src/hooks/session/useDraftMaterialization.ts` hides eager start/draft materialization inside renderer-side session UX. Open question: Should AGH expose draft/start/auth-required/running as explicit daemon lifecycle states, rather than using UI-hidden eager sessions?
- [Direct] `.resources/harnss/src/lib/background/session-store.ts` can reconstruct renderer state from background snapshots, and `.resources/harnss/src/lib/background/agent-store.ts` tracks background agents from events. Open question: What is AGH's canonical durable model for background agents: task rows, event streams, session snapshots, network peer state, or a combination?
- [Direct] `.resources/harnss/shared/lib/codex-rpc.ts` uses request timeouts and process-exit rejection, and `.resources/harnss/electron/src/ipc/acp-sessions.ts` uses initialization/start/auth timeouts. Open question: What timeout and liveness semantics should AGH standardize across local subprocess agents, ACP agents, Codex app-server sessions, and remote/network agents?
- [Direct] `.resources/harnss/src/lib/engine/acp-adapter.ts` maps many ACP update kinds into Claude-like tool names and UI payloads. Open question: Should AGH normalize all provider events into a Claude-like shape, an ACP-like shape, or an AGH-native event schema designed around sessions/tasks/tools/network peers?
- [Direct] `.resources/harnss/electron/src/ipc/claude-sessions.ts`, `.resources/harnss/electron/src/ipc/acp-sessions.ts`, and `.resources/harnss/electron/src/ipc/codex-sessions.ts` each implement separate permission bridges. Open question: Can AGH define one permission/approval abstraction that supports all current providers without losing provider-specific details such as Codex approval policies, ACP options, and Claude tool-use metadata?
