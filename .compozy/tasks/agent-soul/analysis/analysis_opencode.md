# Analysis: opencode

## Scope

- [Direct] This read-only pass covered `.resources/opencode/` only, with emphasis on `.resources/opencode/packages/opencode/src/`, `.resources/opencode/specs/`, `.resources/opencode/.opencode/`, and local instructions in `.resources/opencode/AGENTS.md` and `.resources/opencode/packages/opencode/AGENTS.md`.
- [Direct] The most relevant local instruction constraints are TypeScript/Effect conventions and database/migration conventions in `.resources/opencode/AGENTS.md` and `.resources/opencode/packages/opencode/AGENTS.md`. Those explain why core mechanisms are expressed as Effect services, `InstanceState`, Drizzle tables, and projectors.
- [Direct] No files were edited by the subagent.

## Overview

- [Direct] opencode models identity as a layered runtime context: a project/worktree/sandbox discovered from the current directory, an instance context cached per resolved directory, and sessions tied to project/workspace/parent IDs. Main evidence: `.resources/opencode/packages/opencode/src/project/project.ts`, `.resources/opencode/packages/opencode/src/project/instance.ts`, and `.resources/opencode/packages/opencode/src/session/session.ts`.
- [Direct] State is mostly materialized in SQLite tables (`project`, `session`, `message`, `part`, `todo`, `permission`) defined in `.resources/opencode/packages/opencode/src/project/project.sql.ts` and `.resources/opencode/packages/opencode/src/session/session.sql.ts`. Mutating session operations run through `SyncEvent` definitions and projectors in `.resources/opencode/packages/opencode/src/sync/index.ts` and `.resources/opencode/packages/opencode/src/session/projectors.ts`.
- [Direct] Agent behavior is configured through built-in agents plus discovered markdown/config agents. Built-ins such as `build`, `plan`, `general`, `explore`, `compaction`, `title`, and `summary` live in `.resources/opencode/packages/opencode/src/agent/agent.ts`; project/user agents are loaded from markdown under `{agent,agents}/**/*.md` by `.resources/opencode/packages/opencode/src/config/agent.ts`, with examples in `.resources/opencode/.opencode/agent/triage.md` and `.resources/opencode/.opencode/agent/duplicate-pr.md`.
- [Inference] opencode's "agent soul" equivalent is not one object. It is the combination of project identity, instance context, session row, message/part event history, agent config, permission ruleset, and live status stream.

## Mechanisms / Patterns

- [Direct] Project identity is derived from git state. `.resources/opencode/packages/opencode/src/project/project.ts` walks upward to `.git`, resolves `git rev-parse --git-common-dir`, computes a worktree/sandbox boundary, derives a `ProjectID` from the first root commit when possible, caches it in an `opencode` file under the git common dir, then upserts the project and its sandbox list. Non-git paths fall back to `ProjectID.global` from `.resources/opencode/packages/opencode/src/project/schema.ts`.
- [Direct] Instance identity is an async-local context with `{ directory, worktree, project }`, cached by resolved directory in `.resources/opencode/packages/opencode/src/project/instance.ts`. It exposes path-boundary checks and disposes per-instance resources through `disposeInstance`, while `.resources/opencode/packages/opencode/src/project/bootstrap.ts` eagerly initializes config/plugins/LSP/file watcher/VCS/snapshot and subscribes to `command.executed` to mark a project initialized.
- [Direct] Session identity is durable and hierarchical. `.resources/opencode/packages/opencode/src/session/session.sql.ts` stores `id`, `project_id`, `workspace_id`, `parent_id`, `slug`, `directory`, `path`, `title`, `version`, `summary`, `revert`, `permission`, and timestamps. `.resources/opencode/packages/opencode/src/session/session.ts` creates sessions with generated descending session IDs, relative session paths, parent IDs, workspace IDs, permissions, and `time.created/time.updated`; `fork` clones earlier messages into a new session with remapped message/part IDs.
- [Direct] Session mutations are event-first. `.resources/opencode/packages/opencode/src/session/session.ts` defines `session.created`, `session.updated`, and `session.deleted` as `SyncEvent`s. `.resources/opencode/packages/opencode/src/sync/index.ts` assigns aggregate sequence numbers in an immediate SQLite transaction, invokes a projector, and then republishes through the project bus/global bus. `.resources/opencode/packages/opencode/src/session/projectors.ts` materializes session/message/part changes into SQLite.
- [Direct] The event-log half is conditional. `.resources/opencode/packages/opencode/src/sync/index.ts` only inserts into `event_sequence` and `event` when `Flag.OPENCODE_EXPERIMENTAL_WORKSPACES` is enabled, even though projector mutations always occur. `.resources/opencode/packages/opencode/src/sync/event.sql.ts` defines the event tables.
- [Direct] Agent config is composable. `.resources/opencode/packages/opencode/src/config/config.ts` supports `default_agent`, `agent`, deprecated `mode`, `instructions`, `permission`, `tools`, `mcp`, `command`, `tool_output`, and compaction settings. `.resources/opencode/packages/opencode/src/config/paths.ts` resolves global/project/home/config-dir `.opencode` directories. `.resources/opencode/packages/opencode/src/config/agent.ts` parses markdown frontmatter plus body as agent prompt and normalizes deprecated `tools` into `permission`.
- [Direct] Permissions are rule-based and interactive. `.resources/opencode/packages/opencode/src/permission/evaluate.ts` chooses the last matching `{permission, pattern, action}` rule, defaulting to `ask`. `.resources/opencode/packages/opencode/src/permission/index.ts` loads project-level approved rules, publishes `permission.asked`, waits on a deferred, accepts `once/always/reject`, publishes `permission.replied`, and propagates session-wide rejection.
- [Direct] Tool execution funnels through a common context. `.resources/opencode/packages/opencode/src/tool/tool.ts` defines tool metadata, schema validation, output truncation, and `ctx.ask`. `.resources/opencode/packages/opencode/src/tool/registry.ts` combines built-in tools, local `.opencode/tool(s)/*.{js,ts}` tools, plugin tools, MCP tools, model-specific tool selection, and plugin hooks. Concrete tools ask for permissions in `.resources/opencode/packages/opencode/src/tool/read.ts`, `.resources/opencode/packages/opencode/src/tool/edit.ts`, `.resources/opencode/packages/opencode/src/tool/bash.ts`, and `.resources/opencode/packages/opencode/src/tool/external-directory.ts`.
- [Direct] Task orchestration is session-based. `.resources/opencode/packages/opencode/src/tool/task.ts` creates or resumes a child session via `task_id`, inherits selected parent permission rules, denies `task`/`todowrite` unless the subagent can use them, runs `SessionPrompt.prompt` against the child session, and returns a `<task_result>` plus resumable task ID. `.resources/opencode/packages/opencode/src/session/prompt.ts` also turns command-triggered subtasks into `subtask` parts and handles them through the same task tool path.
- [Direct] Activity tracking is split between transport heartbeat, live run status, and persisted time fields. `.resources/opencode/packages/opencode/src/server/routes/instance/httpapi/handlers/global.ts` emits `server.connected` and `server.heartbeat` every 10 seconds over SSE. `.resources/opencode/packages/opencode/src/session/status.ts` tracks `idle`, `busy`, and `retry` in per-instance memory and emits `session.status`. `.resources/opencode/packages/opencode/src/session/run-state.ts` enforces one active runner per session and flips busy/idle. `.resources/opencode/packages/opencode/src/session/processor.ts` stamps text/reasoning/tool start/end times, retry metadata, and assistant completion time.

## Transferable Patterns

- [Inference] Use an explicit identity envelope for agent-soul state: `{projectID, worktree/root, runtime home, sessionID, parentSessionID, agentID, model/provider, permission profile, timestamps}`. Direct opencode evidence is split across `.resources/opencode/packages/opencode/src/project/project.ts`, `.resources/opencode/packages/opencode/src/project/instance.ts`, and `.resources/opencode/packages/opencode/src/session/session.sql.ts`.
- [Inference] Keep event intent and materialized state separate. opencode's `SyncEvent` plus projectors in `.resources/opencode/packages/opencode/src/sync/index.ts` and `.resources/opencode/packages/opencode/src/session/projectors.ts` are a useful shape for AGH if the TechSpec clearly decides whether the event log is authoritative, an outbox, or only replay support.
- [Inference] Model subagents as first-class child sessions, not anonymous tool calls. opencode's `parent_id` in `.resources/opencode/packages/opencode/src/session/session.sql.ts`, `fork`/`children` in `.resources/opencode/packages/opencode/src/session/session.ts`, and resumable `task_id` in `.resources/opencode/packages/opencode/src/tool/task.ts` give AGH a precedent for traceable delegation and resumable agent work.
- [Inference] Treat permissions as structured policy records evaluated at every tool boundary. opencode's `Permission.Ruleset` in `.resources/opencode/packages/opencode/src/permission/index.ts`, last-match semantics in `.resources/opencode/packages/opencode/src/permission/evaluate.ts`, and tool-local `ctx.ask` calls in `.resources/opencode/packages/opencode/src/tool/read.ts`, `.resources/opencode/packages/opencode/src/tool/edit.ts`, and `.resources/opencode/packages/opencode/src/tool/bash.ts` are transferable to CLI/HTTP/UDS agent-manageable permission flows.
- [Inference] Split liveness from meaningful activity. opencode's SSE heartbeat in `.resources/opencode/packages/opencode/src/server/routes/instance/httpapi/handlers/global.ts` proves the connection is alive; `session.status` in `.resources/opencode/packages/opencode/src/session/status.ts` proves a run is idle/busy/retrying; message/part timestamps in `.resources/opencode/packages/opencode/src/session/processor.ts` prove work happened. AGH should preserve that distinction.
- [Inference] Load custom agents/capabilities from discoverable local files with typed frontmatter and a prompt/body split. opencode does this for agents in `.resources/opencode/packages/opencode/src/config/agent.ts` and for commands in `.resources/opencode/packages/opencode/src/config/command.ts`; AGH can adapt the pattern to its canonical "capability" vocabulary rather than copying opencode's names.
- [Inference] Persist oversized tool output separately and keep the transcript inspectable. opencode saves full outputs to `Global.Path.data/tool-output` through `.resources/opencode/packages/opencode/src/tool/truncate.ts` and `.resources/opencode/packages/opencode/src/tool/truncation-dir.ts`, while returning a preview and file path. This is useful for long-running AGH agent sessions and compaction-safe auditability.

## Risks / Mismatches

- [Direct] opencode's per-instance runtime is TypeScript Effect/`InstanceState` based (`.resources/opencode/packages/opencode/src/project/instance.ts`, `.resources/opencode/packages/opencode/src/effect/instance-state.ts` referenced by many services). [Inference] AGH should not copy this mechanically into Go; it should translate the lifecycle idea into Go contexts, daemon-owned registries, cleanup hooks, and SQLite transactions.
- [Direct] opencode's sync README says the event system was introduced with backward-compatible bus integration, and `.resources/opencode/packages/opencode/src/server/projectors.ts` contains a `convertEvent` compatibility shape for `session.updated`. [Inference] AGH is greenfield alpha, so compatibility shims like converted legacy event shapes are a mismatch unless there is a current protocol contract requiring them.
- [Direct] `.resources/opencode/packages/opencode/src/sync/index.ts` persists event rows only behind `OPENCODE_EXPERIMENTAL_WORKSPACES`, while `.resources/opencode/packages/opencode/src/session/projectors.ts` always mutates materialized tables. [Inference] AGH should avoid an ambiguous half-event-sourced design in the agent-soul TechSpec; choose event log as source, outbox, or audit trail explicitly.
- [Direct] `.resources/opencode/packages/opencode/src/permission/index.ts` loads `PermissionTable` data and appends `always` approvals to in-memory `approved`, while `.resources/opencode/packages/opencode/src/session/session.sql.ts` defines the table. [Inference] Based on local search in `.resources/opencode/packages/opencode/src`, there is no obvious write-back for new `always` approvals, so AGH should define whether "always" is session-only, project-persistent, or user-global.
- [Direct] opencode derives project IDs from git root commits and caches them in git metadata in `.resources/opencode/packages/opencode/src/project/project.ts`. [Inference] AGH should be cautious if agent-soul identity must work for non-git workspaces, generated QA labs, remote sandboxes, or ACP sessions where git root is not stable or not authoritative.
- [Direct] `.resources/opencode/packages/opencode/src/tool/task.txt` says the parent should generally trust agent outputs, and `.resources/opencode/packages/opencode/src/tool/task.ts` returns only a final result plus `task_id`. [Inference] AGH likely needs stronger provenance for autonomous coordination: child session IDs, tool-call event links, permission decisions, and verification evidence should remain queryable through agent-manageable surfaces.
- [Direct] opencode's heartbeat in `.resources/opencode/packages/opencode/src/server/routes/instance/httpapi/handlers/global.ts` is a transport heartbeat, not a persisted lease. [Inference] If AGH needs daemon recovery, detached agents, or multi-agent orchestration safety, the TechSpec should add persisted activity/lease records beyond SSE heartbeat events.

## Open Questions

- [Direct] `.resources/opencode/packages/opencode/src/project/project.ts` uses git-derived project identity. Open question for AGH: should agent-soul identity be git-derived, explicit in `config.toml`, allocated by daemon on project registration, or tied to AGH_HOME/lab roots?
- [Direct] `.resources/opencode/packages/opencode/src/sync/index.ts` supports replay and event sequencing, but event rows are gated; `.resources/opencode/packages/opencode/src/session/projectors.ts` materializes current state. Open question for AGH: is the soul log the source of truth, or is SQLite current state primary with events as audit/outbox?
- [Direct] `.resources/opencode/packages/opencode/src/session/session.sql.ts` stores session-level permission rules, while `.resources/opencode/packages/opencode/src/permission/index.ts` has project-level approved rules. Open question for AGH: which permission scopes exist: agent, session, project, provider, user, runtime, or capability?
- [Direct] `.resources/opencode/packages/opencode/src/tool/task.ts` resumes subagents by passing a child session ID as `task_id`. Open question for AGH: should task resumption use session IDs directly, or a separate task/orchestration ID that maps to one or more sessions?
- [Direct] `.resources/opencode/packages/opencode/src/session/status.ts` keeps live status in memory, while `.resources/opencode/packages/opencode/src/session/processor.ts` persists work timestamps. Open question for AGH: which activity states must survive daemon restart and which can remain ephemeral?
