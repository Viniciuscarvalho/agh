# ADR-001: Verifiable Agent History Continuity

## Status

Accepted

## Date

2026-05-29

## Context

Operators reported that an agent conversation can appear to lose its history after a prompt completes, after reload, or after reopening the session. Even when session events still exist, a blank or misleading chat surface breaks trust because the operator cannot tell whether the agent forgot context, the session failed, or the interface failed to load the record.

This affects more than a visual surface. AGH is agent-first, so an operator and an agent both need a reliable way to inspect whether prior session history is available before continuing work.

## Decision

The MVP will use the "Verifiable continuity" approach: existing session history must remain visible after prompt completion, reload, and reopen, and the visible result must be consistent across Web and CLI surfaces. If AGH cannot load or prove the session history, the user sees an actionable failure state with retry and diagnostic direction instead of a misleading empty conversation.

## Alternatives Considered

### Alternative 1: Web-only restoration

- **Description**: Treat the problem as a single Web conversation rendering defect.
- **Pros**: Smaller product scope and faster visual improvement.
- **Cons**: Does not prove the runtime record is readable outside the Web surface and leaves agents without a reliable manageability path.
- **Why rejected**: AGH features are incomplete if only the Web can operate them.

### Alternative 2: Full conversation recovery suite

- **Description**: Expand the first release into complete session repair, export, replay, and audit workflows.
- **Pros**: Broader trust and richer operator tooling.
- **Cons**: Too large for the immediate user problem and delays the core continuity fix.
- **Why rejected**: The current incident requires continuity visibility first; richer recovery can follow once the baseline is trustworthy.

## Consequences

### Positive

- Operators can trust that an empty chat means no history exists, not that loading silently failed.
- Agents and operators get a shared product expectation for inspecting session continuity.
- Future recovery and audit work can build on a verified baseline.

### Negative

- The MVP must align multiple user-facing surfaces before it can be considered done.
- Some ambiguous empty states need sharper product copy and diagnostic routing.

### Risks

- Users may expect full session repair once diagnostics are visible. The PRD keeps repair workflows out of MVP and focuses on continuity visibility.
