# ADR-001: Verified Marketplace Skill Installation

## Status

Accepted

## Date

2026-05-29

## Context

Operators reported that marketplace skill installation can appear successful while the skill is not clearly visible or usable afterward. That creates a false success path: the operator believes the skill is installed, but the Web surface, CLI surface, or agent discovery path may not prove that the skill is ready for use.

AGH skills affect how agents perform work, so installation cannot be defined only as a completed download or a successful button action. The product must define "installed" as a state the operator and agent can both verify.

## Decision

The MVP will use the "Verified installation" approach: marketplace installation succeeds only when the skill is visible in installed surfaces and usable by eligible agents. Web, CLI, and agent discovery must converge on the same installed state. If AGH cannot verify that state, the user sees a clear reason and next step instead of a false success message.

## Alternatives Considered

### Alternative 1: Web confirmation only

- **Description**: Keep marketplace installation focused on the Web surface and show success after the install action completes.
- **Pros**: Simple user flow and small product surface.
- **Cons**: Does not prove the skill is usable by agents or visible to CLI operators.
- **Why rejected**: AGH skills are agent-operable artifacts, so Web-only success is not a complete product state.

### Alternative 2: Complete skill lifecycle overhaul

- **Description**: Redesign install, update, remove, trust, permissions, and agent assignment in one release.
- **Pros**: Produces a broad and coherent skills management experience.
- **Cons**: Too large for the immediate install-success defect and likely to delay a reliable MVP.
- **Why rejected**: The immediate product gap is false install success; lifecycle expansion can follow after verification is reliable.

## Consequences

### Positive

- Operators get a trustworthy success definition: installed means visible and usable.
- Agents receive the same installed-state expectation as Web and CLI users.
- Failure reports become more actionable because unverified installs are not labeled as successful.

### Negative

- Marketplace install UX must wait for proof before claiming success.
- Some existing success messaging may need to become more conservative.

### Risks

- Users may see more install failures at first because previously hidden verification gaps become explicit. The PRD accepts this as a quality improvement.
